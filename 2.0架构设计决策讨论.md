对。从第一性原理看，**交互式叙事的原子操作就两个**：

```
① Generate（引擎产出内容）
② Receive（引擎接收输入）
```

整个游戏就是这两个操作的序列。序列的编排由 Flow 决定，但操作本身只有这两种：

```
G → G → R → G → R → G → G → R → ...
│   │       │       │   │
序章  行走   玩家回应  GM回应  两段独白  玩家回应

```

所有其他东西——记忆、状态、压缩、Prompt 组装、token 预算——都是 **Generate 内部的实现细节**。它们服务于"如何更好地执行 Generate"，不是独立的顶层步骤。

```
Generate 内部：
  组装上下文（读 State + Memory + Prompt Segments）
  → 调用 LLM
  → 流式渲染
  → 解析状态更新
  → 写入记忆
  → 检查压缩

Receive 内部：
  等待输入
  → 写入记忆
  → 如有 saveToState → 更新状态

分析了 Fenced Block（当前方案）vs Tool Use（新提案）的优劣：

### Tool-Based 方案的优势

1. **结构化保证**：Tool call 有 JSON Schema 校验，不会出现 fenced block 格式错误、YAML 解析失败
2. **时序精确**：GM 可以在叙事中间更新状态，而不是只能在末尾一次性 dump。比如女孩说出第一个词的瞬间就可以 `update_state({girl_language_level: 1})`
3. **查询能力**：GM 可以主动查询记忆 `query_memory("女孩之前说过什么")`，而不是被动依赖引擎注入的上下文
4. **编剧表达力**：编剧可以在 prompt 中写"当玩家做出关键选择时，调用 pin_memory 把这个时刻标记为重要"——比要求 GM 输出特定格式的 block 更自然

### Tool-Based 方案的风险

1. **Token 开销**：每个 tool definition 占 system prompt token，10 个工具可能额外消耗 2-3k tokens
2. **延迟**：Tool call 是 stop-and-resume 模式，频繁调用会让用户体验断断续续
3. **流式渲染复杂度**：需要处理 text → tool_call → tool_result → text 的交错流
4. **LLM 可靠性**：LLM 有时会"忘记"调用工具，或者在不该调用时调用

### 建议：混合方案

```
┌─ 必选工具（引擎自动注入，编剧不需要关心）─────────┐
│  update_state(patch)    — 更新 ScriptState         │
│  signal_input_needed()  — 告诉引擎"该等玩家了"      │
└────────────────────────────────────────────────────┘

┌─ 可选工具（编剧在 prompt 中决定是否启用）──────────┐
│  query_memory(query)    — 查询历史记忆              │
│  pin_memory(content, importance)  — 标记重要记忆    │
│  inject_context(doc_id) — 主动拉取世界知识文档      │
│  advance_flow(node_id)  — 主动跳转流程节点          │
└────────────────────────────────────────────────────┘
```

Generate 操作的变化：
```
当前：
  assemble → LLM(stream text) → parse fenced blocks → update state → render

Agentic：
  assemble → LLM(agentic loop) → {
    text chunk → stream to UI
    tool_call(update_state) → execute → continue
    tool_call(query_memory) → search → return results → continue
    tool_call(signal_input_needed) → break loop
    text chunk → stream to UI
  } → render complete
```

延迟缓解：`update_state` 和 `pin_memory` 是纯本地操作 < 1ms，`query_memory` 约 10-50ms，均可接受。

### 问题一：State Changelog 归属

**场景**：每次 GM 调用 `update_state({girl_language_level: 1})`，除了更新当前值，是否要记录"什么时候、因为什么、从什么值变成了什么值"？
