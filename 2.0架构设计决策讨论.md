对。从第一性原理看，**交互式叙事的原子操作就两个**：

```
① Generate（引擎产出内容）
② Receive（引擎接收输入）

整个游戏就是这两个操作的序列。序列的编排由 Flow 决定，但操作本身只有这两种：

```
G → G → R → G → R → G → G → R → ...
│   │       │       │   │
序章  行走   玩家回应  GM回应  两段独白  玩家回应

所有其他东西——记忆、状态、压缩、Prompt 组装、token 预算——都是 **Generate 内部的实现细节**。它们服务于"如何更好地执行 Generate"，不是独立的顶层步骤。

```
Generate 内部：
  组装上下文（读 State + Memory + Prompt Segments）
  → 调用 LLM
  → 流式渲染
  → 解析状态更新
  → 写入记忆

Receive 内部：
  等待输入
  → 写入记忆

分析了 Fenced Block（当前方案）vs Tool Use（新提案）的优劣：

### Tool-Based 方案的优势

1. **结构化保证**：Tool call 有 JSON Schema 校验，不会出现 fenced block 格式错误、YAML 解析失败
2. **时序精确**：GM 可以在叙事中间更新状态，而不是只能在末尾一次性 dump。比如女孩说出第一个词的瞬间就可以 `update_state({girl_language_level: 1})`
3. **查询能力**：GM 可以主动查询记忆 `query_memory("女孩之前说过什么")`，而不是被动依赖引擎注入的上下文
4. **编剧表达力**：编剧可以在 prompt 中写"当玩家做出关键选择时，调用 pin_memory 把这个时刻标记为重要"——比要求 GM 输出特定格式的 block 更自然

### Tool-Based 方案的风险

1. **Token 开销**：每个 tool definition 占 system prompt token，10 个工具可能额外消耗 2-3k tokens
2. **延迟**：Tool call 是 stop-and-resume 模式，频繁调用会让用户体验断断续续
3. **流式渲染复杂度**：需要处理 text → tool_call → tool_result → text 的交错流
4. **LLM 可靠性**：LLM 有时会"忘记"调用工具，或者在不该调用时调用

### 建议：混合方案

```
┌─ 必选工具（引擎自动注入，编剧不需要关心）─────────┐
│  update_state(patch)    — 更新 ScriptState         │
│  signal_input_needed()  — 告诉引擎"该等玩家了"      │
└────────────────────────────────────────────────────┘

┌─ 可选工具（编剧在 prompt 中决定是否启用）──────────┐
│  query_memory(query)    — 查询历史记忆              │
│  pin_memory(content, importance)  — 标记重要记忆    │
│  inject_context(doc_id) — 主动拉取世界知识文档      │
│  advance_flow(node_id)  — 主动跳转流程节点          │
└────────────────────────────────────────────────────┘
```

Generate 操作的变化：
```
当前：
  assemble → LLM(stream text) → parse fenced blocks → update state → render

Agentic：
  assemble → LLM(agentic loop) → {
    text chunk → stream to UI
    tool_call(update_state) → execute → continue
    tool_call(query_memory) → search → return results → continue
    tool_call(signal_input_needed) → break loop
    text chunk → stream to UI
  } → render complete
```

延迟缓解：`update_state` 和 `pin_memory` 是纯本地操作 < 1ms，`query_memory` 约 10-50ms，均可接受。

三个边界问题详解
问题一：State Changelog 归属
场景：每次 GM 调用 update_state({girl_language_level: 1})，除了更新当前值，是否要记录"什么时候、因为什么、从什么值变成了什么值"？
比如：
Turn 5: girl_language_level: 0 → 1 (女孩第一次发出单音节)
Turn 12: girl_language_level: 1 → 2 (女孩开始说短句)
Turn 18: fall_choice: null → "stay" (玩家选择留下)
为什么这是边界问题：
	•	如果 changelog 是 ScriptState 的一部分：它会被持久化、被存档/读档保存、被跨章继承。好处是完整的状态演变历史随时可查，GM 甚至可以通过 read_state 看到变量是怎么变过来的。坏处是 ScriptState 会越来越大，序列化到 context 里占 token。
	•	如果 changelog 是 独立存储（类似 MemoryState 的一个附属结构）：ScriptState 保持干净，只有当前快照。changelog 独立存在，供 debug 面板查看、供 query_memory 搜索，但不注入到 GM 的 context 里。好处是 ScriptState 精简，坏处是 changelog 和 state 分离后需要额外的同步机制。
	•	如果 不记录 changelog：最简单，但丢失了"状态怎么演变到现在"的信息。对 debug 不利，对 GM 叙事连贯性也可能有影响（GM 不知道变量是什么时候改的）。
核心取舍：changelog 有多少运行时价值（GM 需要看吗）vs 它的 token 成本。

问题二：Prompt Segment 激活状态
场景：编剧写了 20 个 PromptSegment（世界观、角色设定、各阶段的场景描写等）。在运行时，引擎根据 injectionRule 决定哪些 segment 注入到当前 context。那问题来了——"当前哪些 segment 是激活的"这个信息，存在哪里？
具体例子：
	•	PromptSegment "resonance-pool-setting" 的 injectionRule 是 when: state.current_stage == "exploration"
	•	玩家进入 exploration 阶段 → 这个 segment 被激活
	•	玩家离开 exploration → 这个 segment 被停用
为什么这是边界问题：
	•	如果激活状态属于 RuntimeState（临时的、不持久化）：每次 Generate 操作时，PromptAssembler 根据当前 ScriptState 重新计算哪些 segment 该注入。激活状态是计算的结果，不需要存储。好处是简单、无状态、幂等。坏处是每次都要重新 evaluate 所有 injectionRule。
	•	如果激活状态 持久化（存到某个地方）：存档/读档时可以精确恢复"上次加载了哪些 segment"。但这引入了一个问题——如果编剧更新了 injectionRule 但玩家读了旧存档，持久化的激活状态和新 rule 可能冲突。
	•	还有一种情况：GM 通过 tool 主动激活/停用 segment。比如 GM 调用 inject_context("resonance-pool-setting")，手动拉取了一个本来不在 injectionRule 范围内的文档。这次手动激活是一次性的（只影响当前轮），还是持续到被显式停用？
核心取舍：激活状态是"纯函数计算的结果"（从 ScriptState 推导出来，不需要单独存）还是"有独立生命周期的状态"（可以被 GM 覆盖、需要持久化）。

问题三：跨章继承的本质
场景：第一章结束，进入第二章。引擎需要把第一章的部分状态和记忆带入第二章。在之前的模拟中我们设计了：
1. 压缩 ch1 所有剩余原文为最终摘要
2. 合并状态：继承字段保留，重置字段用 ch2 初始值
3. ch2 的 EpisodicMemory 起始状态：entries 清空，summaries 带入 ch1 摘要
为什么这是边界问题：
	•	如果跨章继承是 一次性操作（transition action）：引擎在章节切换时执行一个函数，读 ch1 状态，写 ch2 初始状态，完成后 ch1 的数据可以归档或丢弃。这意味着继承逻辑是过程性的——"执行一次就结束"。好处是简单直接。坏处是一旦执行完，ch1 的原始数据就不可逆地被压缩了，如果玩家想回到 ch1 重玩，需要从存档恢复。
	•	如果跨章继承是 持久化状态（inheritance config + snapshot）：引擎保存一个"继承快照"，记录 ch2 从 ch1 继承了什么。这个快照是 ch2 状态的一部分，可以被查看、被 debug、甚至被 GM 查询（"玩家在第一章做了什么选择"）。好处是可追溯。坏处是增加了状态复杂度。
	•	还有一个子问题：继承规则谁来定？
	◦	编剧在 manifest 中声明 inheritsStateFrom: ch1 + 具体字段列表？
	◦	还是 Architect Agent 自动分析两章的 StateSchema，推断哪些字段该继承？
	◦	还是 GM 在 ch1 结束时通过 tool call 主动选择要传递的状态？
核心取舍：跨章继承是"引擎层面的基础设施操作"（编剧不感知，引擎自动处理）还是"叙事层面的创作决策"（编剧/GM 主动控制传递什么）。

三个问题的共同主题：某个信息到底属于哪个状态类别、是否需要持久化、是引擎自动管理还是 GM/编剧可控。这些边界定义清楚了，四层状态模型（Progress / ScriptState / Memory / Runtime）才算真正稳固。

- 缺点：编剧的原始文件被打碎了，编剧改了原文件后需要重新拆分，拆分结果和原文的对应关系可能模糊

#### 方案 C：虚拟拆分 — Segment = 原文件 + 行号范围引用

```
编剧文件：GM提示词_第二章.md (15000字, 不动)
     ↓ Architect Agent 标记范围
IR：8 个 Segment，每个是 { sourceDoc: "GM提示词_第二章.md", sourceRange: [120, 185], ...metadata }
```

- 优点：原文件不动，粒度细，编辑时可以跳回原文对应位置
- 缺点：编剧改了原文件后行号会漂移，需要重新标记（但这个问题 Architect Agent 重新跑一遍就能解决）

倾向方案 C。理由：
1. 编剧的原文件是 source of truth，不应被引擎改动
2. 但 15k 字全量注入在 token 预算上不现实，必须有更细的粒度
3. 虚拟引用（文件 + 范围）既保留了原文完整性，又实现了细粒度注入
4. 行号漂移的问题本质上等价于"编剧改了文件后需要重新提取"——这在任何方案下都一样
