# RFC: 声明式视觉 IR (Declarative Visual State IR)

> **状态**:Draft / 待评审
> **起草日期**:2026-04-24
> **关联讨论**:`trace和问题记录.md`(v29 Bug A/B/C)、本 RFC 前的实时讨论
> **验证材料**:`scripts/validate-declarative-ir.ts` + `scripts/validate-declarative-ir-out/`(N=5 紧凑 DSL 版本已跑,本 RFC 等待基于新格式再跑 N=30)

---

## §1 动机

### 1.1 现状

视觉切换(场景 / 立绘 / 清场)当前是 **tool call**:

- `change_scene(sceneId)`
- `change_sprite(charId, emotion, position)`
- `clear_stage()`

LLM 在一个 step 里可能先输出一大段 `<d>` / `<n>` 对话旁白,step 结束时才发起这些 tool call。

### 1.2 问题

1. **视觉时序和播放时序错位**。OpenAI-compat 协议里 `content` 和 `tool_calls` 是**并列字段**,没有"先说话后切景"的位置信息—— tool 永远在 step 结束时集中执行。玩家的实际体验是"对白播到一半,场景突然跳",或者"新角色还没出现,就开始说话"。

2. **agentic loop 浪费 request**。如果 LLM 在一个 step 里要切 3 次场景,要么压在 step 末尾批量 call(位置丢失),要么拆 3 个 step(多 2 个 round-trip,TTFT 翻几倍)。

3. **Catch-up / restore 状态反推繁琐**。玩家断线重连或刷新页面时,前端要从 narrative_entries 逆推"当前视觉应该是什么",要按时间顺序把所有 `change_scene` / `change_sprite` / `clear_stage` 重放一遍才能得到当前 stage。任何一条 tool call entry 丢失或顺序错乱(参考 v29 Bug A),stage 就错。

### 1.3 根本原因

视觉状态当前是 **"增量指令"模型**:LLM 发出 diff,前端应用 diff。任何 diff 丢失/乱序 → 最终状态错。

---

## §2 设计原则

1. **每个叙事单元自带完整视觉状态**(declarative,不是 diff)。单元级别的"bg + sprites"就是该单元播放时的 stage 快照。
2. **HTML-like 嵌套语法**,贴近 LLM 训练数据分布,最大化模型直觉服从率。
3. **Silent tolerance > retry**。parser 能吸收大部分 LLM 错误并降级渲染,而不是中断 / 重试。
4. **保留 tool call 通道**作为极端情况 fallback(system prompt 不提,避免混用)。
5. **继承为默认,显式为例外**。省略属性 = 视觉状态继承上一单元。
6. **为 LLM 的"非目标输出"提供合法分类出口,而非 prompt 里禁止**。Langfuse 实际 trace 显示 LLM 会泄漏"让我先读取状态""好的,状态为 INIT,我将执行..."等元叙述;此类内容无对应标签时会溢出到 `<narration>` 污染叙事。与其堆 prompt 禁令(经验上服从率低),不如给一个 `<scratch>` 兜底容器——不渲染、不产出 Sentence、但保留在 context,既消化 LLM 表达冲动,又让 parser 有法可依。凡未来出现"LLM 想说但不该给玩家看"的新类别,优先考虑加新分类出口,而不是新禁令。
7. **代码风格:函数式 / 组合式 / 声明式优先**。parser 层和 IR 推导层避免有状态 class;用纯函数 reducer(`(state, event) → (state, outputs)`)、不可变 state、声明式 schema 描述"哪些标签合法、属性如何映射"。mutation 只在与 ReactFlow / DOM / DB 接线的边界(store / session)出现。

## §3 格式规范

### 3.1 元素清单

| 标签 | 类型 | 说明 |
|---|---|---|
| `<dialogue>` | 容器(顶层) | 对话单元,里面有子 tag + 对话文本 |
| `<narration>` | 容器(顶层) | 叙事单元,里面有子 tag + 叙事文本 |
| `<scratch>` | 容器(顶层) | **LLM 元叙述/内部思考出口**——不产出 Sentence、不渲染、保留给下一轮 context。§2 原则 #6 |
| `<background/>` | 自闭合子标签 | 背景切换(写在 `<dialogue>` / `<narration>` 里) |
| `<sprite/>` | 自闭合子标签 | 单个立绘(同上;可有多个) |
| `<stage/>` | 自闭合子标签 | 清场标记(同上;不能和 `<sprite/>` 同时出现) |

### 3.2 属性

**`<dialogue>`**(容器,可选子: background, sprite\*, stage | stage 和 sprite 互斥):
- `speaker`(必填):说话者 id,manifest 定义
- `to`(可选):受话者 id / id 逗号列表 / `"*"`
- `hear`(可选):旁听者 id 列表
- `eavesdroppers`(可选):偷听者 id 列表

**`<narration>`**(容器,可选子: background, sprite\*, stage):
- 无必要属性

**`<background scene="..."/>`**:
- `scene`(必填):场景 id,manifest 定义

**`<sprite char="..." mood="..." position="..."/>`**:
- `char`(必填):角色 id
- `mood`(必填):情绪 id,manifest 每角色定义
- `position`(必填):`"left"` / `"center"` / `"right"`

**`<stage/>`**(空自闭合):
- 无属性。表意"清场"。

**`<scratch>`**(顶层容器,自由文本,非白名单约束):
- 无属性。
- 行为:**不**产出 Sentence,**不**渲染给玩家,**保留**在 messages-builder 的 assistant 历史里传给下一轮 LLM(in-context 强化)。
- 落 tracing event `ir-scratch`(**非** degrade,是正常事件,只量化频率)。
- 用途:LLM 的规划 / 思考 / 元叙述("让我先读取 state...")写到这里;不得用来藏选项 / 指令 / 工具意图(这些仍必须走工具)。

### 3.3 语法示例

```xml
<dialogue speaker="sakuya" to="mc">
  <background scene="cafe_interior" />
  <sprite char="sakuya" mood="neutral" position="center" />
  抱歉让你久等了,我刚收拾完柜台。
</dialogue>

<dialogue speaker="sakuya">
  <sprite char="sakuya" mood="smirk" position="center" />
  不过,你来得真是时候。
</dialogue>

<dialogue speaker="sakuya" to="mc">
  我看你这几天总来咖啡馆,是在等什么人吗?
</dialogue>
<!-- ↑ 没有 background / sprite / stage 子标签 = 完全继承上一单元 -->

<narration>
  <background scene="street_shadow" />
  <stage />
  你跟着卡琳娜走出咖啡馆。咖啡馆门外一片黑影。
</narration>
<!-- ↑ 切换 bg + 清场 -->

<dialogue speaker="karina" to="mc">
  <sprite char="karina" mood="serious" position="left" />
  <sprite char="sakuya" mood="worried" position="right" />
  这条路上最近不太平,你注意点。
</dialogue>
<!-- ↑ 两个 sprite 子标签 = 同时显示两人 -->
```

### 3.4 视觉状态继承规则

每个单元的 resolved 视觉状态由 parser 前向扫描计算:

**背景** (`bg`):
- 子标签里有 `<background/>` → 用它的 `scene` 值
- 没有 → 继承上一单元的 `bg`

**立绘栈** (`sprites: Sprite[]`):
- 子标签里有 `<stage/>` → 清空为 `[]`
- 子标签里有 1 个或多个 `<sprite/>`(且没有 `<stage/>`)→ **替换**为这些 sprites(不是 append)
- 既无 `<sprite/>` 也无 `<stage/>` → 继承上一单元的立绘栈

**跨 step / 跨 turn 继承**:
- 一个 step 最后一单元 → 下一 step 第一单元 的继承源。**不重置**。
- 玩家输入不是视觉单元,不改变视觉状态。

**初始状态** (playthrough 第一单元):
- parser 初始 `bg=null`,`sprites=[]`
- **第一单元必须显式给 `<background/>`**,否则触发错误(见 §4.4)
- **第一单元不强制给 `<sprite/>` 或 `<stage/>`**,允许"开场只有背景无人物"

### 3.5 白名单约束

所有 id 必须在 script manifest 里预定义:

- `background.scene` ∈ `manifest.scenes[].id`
- `sprite.char` ∈ `manifest.characters[].id`(注意:玩家角色 id 如 "mc" 可以列在 characters 里,如果没列就算未定义)
- `sprite.mood` ∈ `manifest.characters[found].moods[]`
- `sprite.position` ∈ `{"left", "center", "right"}` (硬编码)
- `dialogue.speaker` / `dialogue.to` / `dialogue.hear` / `dialogue.eavesdroppers` ∈ `manifest.characters[].id ∪ {"*"}` (对 to/hear/eavesdroppers)

## §4 Parser 设计

### 4.1 底层:从手搓状态机换到 `htmlparser2`

**现状**:`src/core/narrative-parser.ts` 是 ~350 行手搓流式状态机,处理 `<d>` 一种容器 tag。

**问题**:新设计引入嵌套(`<dialogue>` 内有 `<sprite/>` 等子标签),手搓状态机要扩展到处理嵌套,复杂度陡增。

**方案**:换 `htmlparser2`:
- 真流式:`parser.write(chunk)` / `parser.end()`
- 事件驱动:`onopentag(name, attrs)` / `ontext(text)` / `onclosetag(name)`
- 容错:坏标签不抛,降级为文本
- xmlMode 可收紧语法
- 体积小,~40KB min

**保留的业务逻辑**(不是 XML parser 职责):
- Sentence 组装(open dialogue → 新建 buffer,close → emit)
- 视觉继承前向扫描
- Narration flush 时机(`trim().length > 0` 才 emit,打字机友好)
- `\n\n` 不 trim(下游段落切分器依赖)
- `endsWith('<')` 跨 chunk safe-len 裁剪
- Stream 被切断时 force-close + `truncated:true`(见 §4.4)

### 4.2 事件 → Sentence 映射

```typescript
interface Sentence {
  kind: 'dialogue' | 'narration' | 'signal_input' | 'scene_change';  // scene_change 保留给兼容
  text: string;

  // 视觉状态(resolved,已应用继承)
  bg: string | null;
  sprites: Sprite[];

  // 派生:和前一 Sentence 比
  bgChanged: boolean;
  spritesChanged: boolean;

  // 对话专属
  speaker?: string;
  addressee?: string[];
  overhearers?: string[];
  eavesdroppers?: string[];

  // 截断标记(stream 被切断且该单元未闭合)
  truncated?: boolean;
}

interface Sprite {
  char: string;
  mood: string;
  position: 'left' | 'center' | 'right';
}
```

**事件翻译**:

- `onopentag('dialogue', attrs)` → 新建 pending unit,`kind='dialogue'`,记 speaker/to/hear/eav
- `onopentag('narration', attrs)` → 新建 pending unit,`kind='narration'`
- `onopentag('background', attrs)` → 记到 pending unit 的 `pendingBg = attrs.scene`
- `onopentag('sprite', attrs)` → 记到 pending unit 的 `pendingSprites.push({...})`
- `onopentag('stage', attrs)` → 记到 pending unit 的 `pendingClearStage = true`
- `ontext(text)` → 如果在 dialogue/narration 内,累积到该单元的 text buffer;否则丢弃(容器间的空白)
- `onclosetag('dialogue' | 'narration')` → resolve 视觉状态 → emit Sentence → 清 pending

### 4.3 Silent Tolerance 错误处理

这是本 RFC 最核心的设计决策。**所有 LLM 错误都尽可能被 parser 吸收,不触发 retry**。

**语法层**:

| 错误 | 处理 |
|---|---|
| `<dialogue>` 未闭合(stream 截断) | `finalize()` 强制 close,emit Sentence 且 `truncated: true` |
| 容器内嵌套容器(`<dialogue>` 套 `<narration>`) | 外层 close 时先 emit 外层(用前半内容),内层单独 emit |
| 未知顶层标签(`<scene>`、随手写的 `<div>` 等) | 丢弃该标签及其内容,不影响其他 |
| 容器之外的裸文本 | 降级为 narration(向后兼容当前行为) |
| `htmlparser2` 内部错误(极罕见) | 触发 retry(上限 1 次,这是 retry **唯一**合法触发条件之一) |

**语义层(Sprite)**:

| 错误 | 处理 |
|---|---|
| `<sprite/>` 缺必填属性(char/mood/position 任一缺) | 整个 sprite drop,其他 sprite 保留 |
| `char` 不在 manifest | sprite drop |
| `mood` 不在该 char 的 moods 白名单 | sprite drop(**不**降级到 "neutral",降级会掩盖 prompt 漂移) |
| `position` 非 left/center/right | sprite drop |
| 两个 `<sprite/>` 同 `char` 不同 `position` | 取最后一个(LLM 改主意了) |
| 两个 `<sprite/>` 同 `position` 不同 `char` | 取最后一个 |
| `<stage/>` 和 `<sprite/>` 同时出现 | 以 `<stage/>` 优先(清场),所有 sprite 丢弃,记 warn |

**语义层(Background)**:

| 错误 | 处理 |
|---|---|
| `<background/>` 缺 `scene` | drop 该 background,继承上一帧 bg |
| `scene` 不在 manifest | drop,继承上一帧 bg |
| 多个 `<background/>` 在同一单元 | 取最后一个 |

**语义层(Dialogue 属性)**:

| 错误 | 处理 |
|---|---|
| `speaker` 缺失 | 该 dialogue 降级为 narration(纯文本,无 speaker) |
| `speaker` 不在 manifest | 保留,UI 显示 "Unknown Speaker"(LLM 可能编 NPC 名字,保留原文比丢掉更尊重叙事) |
| `to` / `hear` / `eav` 里的 id 不在 manifest | **保留**(同上,LLM 可能编 NPC,不硬删)。**🔴 待确认** |

**初始状态违规**(特殊):

| 错误 | 处理 |
|---|---|
| playthrough 第一单元没 `<background/>` | 触发 retry(上限 1 次,retry **唯一**第二个合法触发条件) |

**所有 silent-degrade 都要记 warn 到 tracing**(见 §10)。

### 4.4 Stream 截断兜底

LLM `finishReason='length'` 时,stream 可能在任何位置被切断:

- 最后一个容器未闭合 → `finalize()` force-close,Sentence 带 `truncated: true`
- 属性值中间切断(如 `<sprite char="saku`) → drop 该 sprite tag,容器继续
- 标签名中间切断(如 `<spri`) → drop

**配合续写机制**(见 §8)消化大部分截断情况。

## §5 与现有系统的接线

### 5.1 `scenePatchEmitter` 改造

**现状**:`src/core/game-session.ts` 里 `scenePatchEmitter` 在 `change_scene` / `change_sprite` / `clear_stage` tool 执行时触发,emit 一个 WS `scene-change` 事件到前端,同时在 Sentence 流里插入一个 `scene_change` 类型的 Sentence。

**新方案**:

- **tool 实现保留不删**(§9),但生产路径不应该触发
- parser 每 emit 一个 Sentence 时,**Sentence 自带 resolved 视觉状态**(`bg` / `sprites`),前端直接从 Sentence 读
- **砍掉独立的 WS `scene-change` 事件和 Sentence 流里的 `scene_change` 单元**
- catch-up 时前端从 Sentence 序列直接读取当前 visible sentence 的 `bg` / `sprites` 即可,无需反推

### 5.2 前端消费更新

**VNStage / VNStageContainer**:

- 不再消费独立 `scene-change` WS 事件
- `bg` / `sprites` 直接从 `parsedSentences[visibleSentenceIndex]` 取
- Transition(淡入淡出等)由前端自己比对相邻 Sentence 的 `bgChanged` / `spritesChanged` 决定

**game-store `appendSentence` / `advanceSentence`**:

- 继续 skip `signal_input` 类型
- 原来 skip `scene_change` 的逻辑可删(新格式没这种类型)

### 5.3 messages-builder 不变

LLM 看到的 assistant message 就是它自己原文(含 `<dialogue>...</dialogue>` 标签)—— in-context 强化靠的是"看自己上轮怎么写的",天然延续格式。

### 5.4 WS 协议变更

**新增**:无

**删除**:`scene-change` 事件

**兼容**:在 `protocolVersion` 为 v1 的老 playthrough 上继续发 `scene-change`(§6)

## §6 迁移 / 兼容

### 6.1 Script 版本标记

Script manifest 增加字段:

```typescript
interface ScriptManifest {
  // ... 现有字段
  protocolVersion?: 'v1-tool-call' | 'v2-declarative-visual';  // 缺省 = 'v1-tool-call'
}
```

### 6.2 Session 启动分叉

`sessionManager.getOrCreate` 读取 manifest 的 `protocolVersion`:

- `v1-tool-call` → 走现有 system prompt + 现有 parser + 现有 tool call 视觉通道
- `v2-declarative-visual` → 走新 system prompt + 新 parser + declarative 视觉通道

### 6.3 老 playthrough 不迁移

- v1 playthrough 的 narrative_entries 保持原样,下次玩继续用 v1 parser
- v2 playthrough 从第一个 entry 就是新格式,parser 用新路径
- 两种 entry 在 DB 里共存,playthrough 层按 script 的 protocolVersion 决定消费方式

### 6.4 编剧升级路径

**🔴 待定**:是否给编剧一个"升级这个 script 到 v2"按钮?

**推荐**:V2 上线初期**不做 UI**。编剧想升级就手动改 manifest 里 `protocolVersion` 字段(后续编辑器 UI 可见)。等有 3+ 个剧本升级经验再做自动化。

## §7 System Prompt 规格

基于 N=5 验证过有效 + N=30 失败分析后的修订版模板。**注意:新格式 N=30 未跑,这只是预期配方,以验证结果为准。**

```
你是一个互动视觉小说 (VN) 的叙事生成器。你的输出格式是结构化 XML 标签。

[格式定义]
每段叙事由若干单元组成,每个单元是以下两种之一:

<dialogue speaker="<角色id>" to="<受话者id>" [hear="..."] [eavesdroppers="..."]>
  [子标签,可选]
  对话文本
</dialogue>

<narration>
  [子标签,可选]
  叙事文本
</narration>

[视觉子标签]
<background scene="<场景id>" />      ← 切换背景
<sprite char="<角色id>" mood="<情绪>" position="left|center|right" />  ← 显示单个立绘
<stage />                             ← 清除所有立绘

[视觉继承规则]
- 省略 <background/> = 继承上一单元的背景
- 单元里出现任意 <sprite/> = 替换整个立绘栈(不是追加)
- 单元里出现 <stage/> = 清除所有立绘(和 <sprite/> 互斥)
- 单元里既无 <sprite/> 也无 <stage/> = 立绘栈完全不变

[内部思考 / 元叙述出口 — <scratch>]
如果你想在输出里夹带"让我先查一下状态""按规则我现在应该..."之类
**给自己看**的思考或元叙述,请把它包在 <scratch> 顶层容器里:

<scratch>
现在是开场第一单元,我需要先调 read_state 拿到玩家名字。
</scratch>

<scratch> 不会渲染给玩家,但会保留在下一轮对话历史里。
不要把 <scratch> 用来藏选项、工具调用或 scene 数据——那些仍然必须走
正式标签或工具。<scratch> 只装"我的思考草稿"。

[白名单 - 严禁编造]
场景:{scene_ids_from_manifest}
角色:{char_ids_from_manifest}
每角色情绪:{char_moods_from_manifest}
位置:left / center / right(只此三档)

[输出纪律 - 硬性禁止]
- 不要写 markdown 代码块(```)
- 不要写解释性旁白("接下来我将..."、"好的,状态是 X")——这类元叙述请写进 <scratch>
- 不要写其他标签(<scene>, <choice>, <bgm>, <div> 等)
- 不要在容器之外(裸文本)写任何文字 —— 所有文字必须在 <dialogue> / <narration> / <scratch> 里

[输出预算]
本轮可用输出预算:{maxOutputTokens} token。
请自行控制单元长度。如预算接近耗尽(估计剩余<500),
立刻闭合当前标签,然后调用 signal_input_needed tool。

[Few-shot 续写示例]
{8-unit example in the target script's voice}

[特殊规则]
- playthrough 第一个单元必须写 <background/>,否则无法渲染
- 想不出具体细节(如某人的情绪)时,省略对应属性/子标签即可,不要填 "?" 或占位符
```

### 7.1 关键修订(对比 N=30 失败分析)

| N=30 错误模式 | N=30 出现 | 新 prompt 应对 |
|---|---|---|
| `mc` 误当立绘 id | 6 次 | 新格式允许 `<sprite char="mc" .../>`;manifest 决定是否渲染。**prompt 不再禁 mc**。 |
| `?` 占位符 | 3 次 | prompt 明文"想不出就省略,不要填 `?`"。新格式的"省略=继承"让 LLM 有自然出路。 |
| 编造 `old_man` | 1 次 | 白名单前的"严禁编造"在 prompt 开头加粗 |
| 未闭合截断 | 2 次 | 预算提醒 + §8 续写 |
| DSL 语法错(`sp="sp"`、重复 id) | 2 次 | **新格式没有 DSL,这类错误归零** |
| 元叙述泄漏到 `<narration>`("让我先读取状态...")| 实际 trace 观察 | **新增 `<scratch>` 合法出口**——从 prohibition 转为 classification,预期 LLM 服从率显著高于禁令 |

## §8 maxOutputTokens 截断处理

三层:

### 8.1 Prompt 预算提醒

见 §7 `[输出预算]` 段。成本几乎零。效果估计:略微抑制 LLM 贪写长单元的倾向。

### 8.2 续写机制(扩展现有 post-step 补刀)

**现状**:`src/core/llm-client.ts` 已有一个"post-step 补刀"机制(line 562+),原设计是为"LLM 忘了调 signal_input_needed"。主 generate 结束时若未 signal,自动发 follow-up request。

**扩展**:把触发条件从"没 signal"扩到"没 signal **或** 最后一单元被截断":

```typescript
// llm-client.ts - post-step 触发条件扩展
const needsFollowup =
  !calledSignalOrEnd ||
  (lastStep.finishReason === 'length' && lastSentenceTruncated);

if (needsFollowup) {
  const reason = lastSentenceTruncated
    ? '上次输出因 token 预算耗尽被截断,最后一个单元未闭合。请接着截断点续写,先闭合未闭合的标签,然后正常结束 / 调 signal_input_needed。'
    : '主 generate 结束时未调用 signal_input_needed 或 end_scenario。请决定本轮应该如何收尾。';

  await runFollowupStep({ systemExtraNote: reason, ... });
}
```

**好处**:不是 retry(不重写),是续写(接着写)。延迟只多一个 round-trip,内容零浪费。对用户透明(前端看到的是一个连续的 stream)。

### 8.3 Parser truncated 兜底

即使没有续写机制也不崩:最后一单元 `truncated: true`,UI 显示半句话(可视觉上加个"..."),然后 signal_input 接管。

## §9 Tool Fallback 通道

### 9.1 保留什么

代码层面继续保留:
- `change_scene(sceneId)` tool
- `change_sprite(charId, emotion, position)` tool
- `clear_stage()` tool
- 对应的 `scenePatchEmitter` 逻辑

### 9.2 System prompt 不提

v2 protocolVersion 的 system prompt 里**完全不提**这些 tool。LLM 只看到 IR 路径是唯一输出方式。

### 9.3 什么情况会触发

- 极边缘的 agentic loop edge case:parser 完全无法解析某段输出 + retry 也失败
- 引擎内部如有需要硬性切换 stage 的场景(目前无,保留扩展)

### 9.4 为什么不直接删

- **逃生口**:出现线上未预料的 LLM 漂移(比如它决定全部写 JSON),至少能 flip 一个 bit 把 prompt 切回 tool 模式
- **迁移期**:v1 playthrough 还在跑老 tool call,代码不能全删

## §10 Tracing & 可观测性

### 10.1 Silent-degrade 分类

每个 silent-degrade 在 Langfuse / tracing 里打 tag:

- `ir-degrade:sprite-unknown-char` / `ir-degrade:sprite-unknown-mood` / `ir-degrade:sprite-invalid-position`
- `ir-degrade:sprite-missing-attr`
- `ir-degrade:stage-and-sprite-conflict`
- `ir-degrade:bg-unknown-scene`
- `ir-degrade:bg-missing-attr`
- `ir-degrade:unknown-toplevel-tag`
- `ir-degrade:container-truncated`
- `ir-degrade:initial-no-background`(初始状态违规触发 retry 用)

**正常分类事件**(非 degrade,只量化频率,不告警):

- `ir-scratch`:本轮 LLM 产出了一个或多个 `<scratch>` 块。metadata 记录 `count` / `totalChars` / `turn`。
  - 用途:量化"元叙述泄漏转移率"——如果 `<scratch>` 频次很高、`<narration>` 里的"好的,状态..."一类旁白显著下降,说明分类出口生效;反之则需要修正 prompt。
  - 单独命名而非 `ir-degrade:*` 的理由:**按设计这是 LLM 的合法表达**,不是降级,按 degrade 归类会在告警面板误导。

### 10.2 用途

- 后台看 `ir-degrade:*` 分布,识别高频错误模式
- 高频错误模式 → system prompt 加针对性强化 / few-shot 加反例
- 低频错误 → 吸收即可,不值得调 prompt

### 10.3 告警阈值

**🔴 待定**:每 1000 turn 的 `ir-degrade:*` 总和超过 X 就告警?具体 X 值等生产数据定。

   - 命名待定:
     - `<thinking>` — Anthropic 推荐,但 DeepSeek-R1 用 `<think>` 做 reasoning,V3 chat 可能混淆
     - `<scratch>` — prompt engineering 圈"scratchpad"的直接对应,无冲突风险

   - **推荐**:`<scratch>`(无冲突 + 强先验)。
   - **行为**:
     - 顶层容器,和 `<dialogue>` / `<narration>` 同级
     - 自由文本,不受白名单约束
     - **不**产出 Sentence,**不**渲染给玩家
     - **保留**在 messages-builder 给下一轮 LLM 看(in-context 强化)
     - 落 tracing tag `ir-scratch`(**非** degrade,是正常事件)

## §14 附录:设计演进记录

| 阶段 | 方案 | 放弃原因 |
|---|---|---|
| 初始 | `change_scene` / `change_sprite` tool call | 视觉时序和播放时序错位,catch-up 复杂 |
| 方案 A | 单行 inline command 标签 `<scene/>` `<sprite/>` `<clear/>`(类似 XML self-closing) | 不解决"位置对齐"根本问题,仍然是增量指令 |
| 方案 B | 紧凑 DSL 属性 `sp="sakuya:smile@c"` `bg="cafe"` 写在 `<d>` / `<n>` 上 | N=30 验证四项通过率只有 63%,长上下文 30%,DSL 语法对 LLM 不直观 |
| **方案 C(本 RFC)** | HTML-like 嵌套子标签 + 全名 + declarative state | 预期 LLM 直出通过率显著提升,silent tolerance 吸收剩余错误 |

### §14.1 为什么选 XML 而不是 DSL(面对行业反证)

**反证**:VN / AVG 行业主流引擎(Ren'Py、NaniScript、KAG、Ink、Yarn Spinner)**全部选择自家 DSL 而非 XML**,纯 XML 在叙事领域被**普遍嫌冗长**。这是明确的行业信号,需要正面回答:我们为什么选 XML?

**回答**:这些 DSL 是为**人类编剧手写**设计的优化目标——节省按键、自然语言顺序、缩进即结构。但本项目**不是给编剧手写的格式**,是**给 LLM 生成的中间表示**。优化目标完全不同:

| 维度 | 人类手写 | LLM 生成(本项目) |
|---|---|---|
| 冗余成本 | 按键次数、视觉密度 | 输出 token 预算(实际每单元多 20-40 token,可接受) |
| 语法习得难度 | 需要学习 DSL 规则 | LLM 训练数据分布决定,**XML/HTML 分布远大于任何 VN DSL** |
| 容错要求 | 编辑器高亮 + 保存前 lint | 必须宽松 parse + silent tolerance |
| 结构化处理 | IDE 可以做 AST 工具链 | 需要被机器解析 → IR → 渲染 |

**结论**:行业 DSL 选择**不适用于 LLM-generated 场景**。XML 在 LLM 生成稳定性上的优势来自两方面证据:

- **Anthropic 官方推荐**(`platform.claude.com/docs/build-with-claude/prompt-engineering/use-xml-tags`):"XML tags help Claude parse complex prompts unambiguously... Wrapping each type of content in its own tag reduces misinterpretation."(有一手出处可引)
- **本项目 N=30 采样经验**:方案 B(紧凑 DSL)63% 四项通过率 → 方案 C(XML 嵌套)87%。本地实测数据,**非跨模型断言**。

(原稿此处曾声称"DeepSeek/Qwen 对 XML 嵌套服从度 > 嵌套 JSON",**该断言无一手文档支撑,已删除**。我们只能诚实陈述"对 DeepSeek 本项目样本"的结论,不能推广到其他模型。)

### 15.2 Prior Art(真实参照,注明"取什么 / 不取什么")

**TEI (Text Encoding Initiative)** — 人文学科 XML 大标准
- **启发**:TEI 的 `<said who="...">` `<sp>` `<stage>` 三个标签就是我们 `<dialogue speaker>` `<dialogue>` `<narration>` 的直接语义对应。几十年的文献标注实践证明这组语义在描述戏剧/小说叙事上**是 sufficient 的**,我们不是发明新范畴。
- **不取**:TEI 标签集庞杂(含文献学、版本学、手稿标注),学术气重,非给生成任务设计。本 RFC 只借用三元组语义,不引入 TEI 命名空间。

**SAIBA / BML (Behavior Markup Language)** — 虚拟角色多模态标记(2000s 研究)
- **启发**(**来自背景知识,未在本次研究中拿到一手 URL**——mindmakers.org 本次 ECONNREFUSED、ResearchGate 403、SAIBA PDF 404):据学界公开讨论,BML 在一个 speech act 里挂行为子标签(类似 gesture/face/gaze)作为兄弟元素。若属实,这和我们 `<sprite/>` `<background/>` `<stage/>` 嵌在 `<dialogue>` 里的拓扑同构。
- **免责**:具体元素名和规范细节**未一手验证**,需要后续查阅 Kopp et al. 2006 原论文补充。本 RFC 不依赖 BML 做硬设计决策,仅列此作后续深入方向。
- **替代支撑**:TEI 的 `<stage>` 嵌在 `<sp>` 里提供了同样的"per-speech 行为标记"模式(**有一手规范**),作为我们的权威对齐已够用,BML 可降为"锦上添花"。

**articy:draft (`.articyx` XML)** — 商业 AVG 对话树工具,Unity/Unreal 大量使用
- **启发**(有一手 XSD 确认):articy 的 `FlowFragmentType` / `DialogueFragmentType` / `HubType` / `JumpType` 直接对应节拍 / 对话 / 分叉,`DialogueFragmentType` 有 `Speaker`(IdRef 引用)和 `StageDirections` 字段——和我们 `<dialogue speaker>` + 子 `<stage/>` / `<background/>` 直接同构。
- **扩展机制值得借鉴**:articy 的 `FeatureType` 用 `PropertiesType` 的 `xs:choice`(LocalizableText / Number / String / Boolean / NamedReference / References / Enum),通过 `IdRef` 解耦定义和使用。**对本 RFC §13 将来加 `<internal_state_delta/>` 等字段有直接参考价值**:可以学 articy 的模式给 `<dialogue>` / `<narration>` 加一个可选 `<feature name="..." value="..."/>` 通用扩展子标签,而不是每次新加具象 tag。每加具象 tag 都要更新 parser + system prompt + whitelist;用 Feature 扩展只要注册字段定义。
- **不取**:articy 是**节点图**(有 Jump/Hub/跳转),我们是**线性流**(LLM 生成顺序序列)。图模型不适合流式 LLM 输出。

**FDX (Final Draft XML)** — 好莱坞剧本行业事实标准
- **启发**(**业内惯例,本次未从 Final Draft 官方规范一手确认** —— 官方 KB socket 断开、多个 GitHub 镜像 404。Fountain 给出的屏幕写作 paragraph 分类是:Scene Heading / Action / Character / Dialogue / Parenthetical / Transition / Shot / Dual Dialogue / Lyrics / Centered,但 Fountain 明说"不解释 FDX 内部 XML 架构"):据广泛流传,FDX 用 `<Paragraph Type="...">` 单一容器 + type 属性区分。我们**没采用但认真考虑过**这种单 `<beat type=...>` 的替代设计。
- **不取原因**:单 `<beat type=...>` 对 LLM 更抽象,训练数据里 `<dialogue>` 这种**具象 tag 频次远高于** `<beat type="dialogue">`。选**具象 tag**押注 LLM 直觉服从率。
- **Parenthetical 值得注意**:Fountain/FDX 有 Parenthetical(`(冷笑)`)表达"说话方式修饰",TEI 里对应 `<stage type="delivery">`。**我们 IR 里无对应**——见 §13 Open Q6。

**TTML / SMIL** — W3C 时基多媒体 XML
- **启发**:`<p begin="..." dur="...">` 为每拍加时间戳,适合带配音 / 自动推进的 AVG。
- **不取**:本项目是玩家点击推进(无预定时序),时间戳没意义。**但将来如果加"自动播放 / 配音同步",可以向 TTML 方向扩展** `<dialogue dur="3s">`——本 RFC 的 `<dialogue>` 留了命名空间给这个扩展。

**Ren'Py / NaniScript / KAG / Ink / Yarn Spinner** — 主流 VN DSL
- **反例**:§14.1 已分析,这些是**人类手写**优化,不适用 LLM 生成。**不取任何设计**,但它们的存在本身是重要反证,必须显式回答为什么不学它们。

**Inform 7 / TADS / Scheherazade / Drammar / MPML / VHML / APML / EMMA** — 学术叙事标记
- **已知存在,未进一步研究**。本 RFC 不引用。列在这里仅作后续深入调研的入口(如果某次评审需要更多 prior art 对齐)。

### 15.3 设计归属声明

- 基础嵌套结构 + "视觉状态作为子标签"的设计思路:**本项目独立推导**(N=30 方案 B 失败 → 讨论中由用户提出嵌套 tag 方向)
- "speaker/said 语义三元组":对齐 TEI 的已证范畴,不声称原创
- "视觉行为作为 speech act 子标签":和 BML 同构,不声称原创
- "silent tolerance 错误处理策略":本项目独立推导(基于 N=30 失败分布分析)
- "continuation 机制处理截断":扩展项目已有 post-step 补刀机制(2026-04-24 commit)

**本 RFC 的独创性**集中在:(1) 视觉状态的**继承**规则;(2) silent tolerance 的**分级策略**;(3) 和 LLM 续写机制的集成。其他结构决策在已知 prior art 范围内。
