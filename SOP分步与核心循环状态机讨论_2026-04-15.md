## Assistant

这是个很实在的问题。在 agentic loop + tool calling 的场景里（就是你们引擎现在的架构），有几层可以叠加用的技巧，从强到弱：

### 1. 工具化 step 边界（最强约束）

引擎已经有 tool calling 能力，把 SOP 的每个关键阶段定义成一个**必须调用的工具**，比模型自己想着走流程可靠得多。

例如 MODULE7 如果是"开场 → 探索 → 冲突 → 收尾"这种，可以设计：

```
enter_phase(phase: "opening" | "exploration" | "conflict" | "closing")
```

并在 engine-rules 里写：
- 开局后**必须**先调用 `enter_phase("opening")`
- 不调用 `enter_phase("exploration")` 不准触发战斗工具
- `end_scenario` 前必须处于 `closing`

再把 `currentPhase` 写入 STATE section，每轮 context 里暴露给模型，它就回不了头。

### 2. 显式 state machine + gated tools

比 #1 弱一点：不强制 advance tool，但在 tool-executor 里做**门控**——某些工具只有在特定状态下才允许调用（不满足直接 reject，返回错误让模型重试）。

这个对"SOP 不能跳步"非常有效，因为失败会被模型看到并纠正。

### 3. 结构化 SOP 块（Prompt 工程层面）

MODULE7 如果现在是散文式的，最基本的工程化是：

```markdown
# SOP: 场景开场

## 前置条件
- state.phase == "pre_opening"
- player 尚未发言

## 执行步骤
1. 先 narrate 环境描写（100-200 字）
2. 介绍 NPC 出场（1-2 人）
3. 调用 signal_input_needed，hint="你想..."

## 完成标志
- narrate 输出 ≥ 100 字
- 至少 1 个 NPC 被 mention
- signal_input_needed 已调用

## 禁止
- 不要提前推进到冲突
- 不要一次写完整章
```

关键是**前置条件 / 执行步骤 / 完成标志 / 禁止**四段式。模型不是不会按 SOP 走，是原始散文里步骤边界太模糊。

### 4. Few-shot 完整示例

在 prompt 里放 1-2 个**走完整条 SOP 的完整对话样例**（含 tool call 序列），比十倍长度的规则描述都管用。特别是开头那几步，示例一放就知道该怎么开局。

### 5. 每轮 reminder（轻量）

在 context-assembler 里拼一段：
```
## 当前状态
- 你处于 SOP 第 2 步（探索阶段）
- 还需完成：[ ] 介绍地点 A，[ ] 触发线索 X
```

动态从 state/memory 推出来。模型每轮都被提醒"你在哪、还差什么"。

### 二、改造建议（按优先级）

#### ★★★ 1. 阶段机显式化 + 分阶段 prompt 注入

引擎已经有 `context-assembler` + virtual sections，这是最关键的改造：

**把 state 字段正式化**（写进 IR 的 stateSchema）：
```ts
{
  chapter: 1 | 2,
  phase: 1 | 2 | 3 | 4 | 5,
  route: 'mainline' | 'cold' | 'very_cold',
  sceneLevel: 'A' | 'B' | 'C',
  girlLanguageLevel: 0 | 1 | 2 | 3 | 4,
  playerArchetype: 'A' | 'B' | null,
  // ...
}
```

**把各阶段详细指令拆成独立文档**（editor 里一个 doc 一个阶段），通过 virtual section `current_phase_detail` 动态注入：

- assembly 时读 `state.chapter` + `state.phase`
- 只拼**当前阶段**的 D/E/I 节内容进 context
- system prompt 常驻部分只留：GM 角色 + 硬规则 + B 节世界观 + 工具用法 + 硬性 self-check

**收益**：system prompt 从 1088 行降到 ~350 行常驻 + 150-250 行切片。每轮模型看到的"当前该做什么"权重从 5% 升到 40%+。指令遵循率会质变。

#### ★★★ 2. 阶段推进工具化 + checkpoint tool

当前"阶段转换"只靠模型自觉"D 节说到轮次预算"。改成硬约束：

```ts
// 引擎内建
advance_phase({ to: 'ch1-phase-3', reason: string })

// 章节特定 checkpoint
trigger_first_speech({ word: string, trigger: string })  // ch1-phase-5 必须调用
confirm_accompany_intent({ method: 'position' | 'action' | 'language' })  // ch1 收尾前必须
select_entry_route({ route: 'active' | 'passive' | 'independent' })  // ch2-phase-2
operate_crystal_seal({ action: 'remove' | 'leave' })  // ch2-phase-4 可选
```

`engine-rules.ts` 里写：
- `advance_phase` 到 `ch1-phase-5` 前，必须已调用 `trigger_first_speech`
- `end_scenario` 前，必须已调用 `confirm_accompany_intent`（或 route 转为 cold）

工具门控做在 `tool-executor` 里：前置条件不满足时 reject 并返回错误，模型会收到反馈并纠正。这比散文"必须"硬得多。

#### ★★ 3. 硬规则引擎侧强制校验

专有名词禁令（硬规则一）、字数分级（硬规则二）完全可以从 prompt 移到引擎：

- **专有名词**：在 narrative 写入前扫描黑名单（苗圃 / 孵化所 / 共鸣池 / 涅瓦 / 卡伦 / 死海 / 大泽 / 马里亚纳 / NEVA / CAREN），命中就 reject 并要求重写
- **字数**：narrative 写入时根据当前 `state.sceneLevel` 校验字数，超限截断或重写

这些是**机械判定**，用 LLM 自查是浪费 token 还不可靠。放引擎一次写定，永远生效。

#### ★★ 4. 自查清单 23 条分层

当前 J 节是 post-hoc list，实际模型不一定每轮都扫。分成三类：

| 类别 | 处理 |
|------|------|
| 机械可判（专有名词、字数、重复描写） | 引擎校验，prompt 里删 |
| 行为门控（同行意图、第一次开口、路径确认） | 变成 checkpoint tool（见 #2） |
| 质感判断（情绪命名、散文质地） | 留在 prompt，但压缩到 5-8 条 |

23 条 → 留 6-8 条软性提醒。

#### ★ 5. 世界观（B 节）改成 memory summary

B 节现在是 600+ 字的世界观全景，每轮都在 system prompt 里。但它其实不需要每轮完整看到——可以：

- 放进一个永久的 memory summary（`category: 'worldview'`），assembly 时作为 MEMORY section 输出
- 或者按阶段相关性切片：第一章只放 B.1，进入第二章 phase-1 时才注入 B.2

#### ★ 6. 去冗余

快速扫了一遍，以下明显重复：
- 硬规则一 / J.11 / 阶段三的"不说穿"——专有名词禁令说了 3 遍
- A 节和 D 节都在说"GM 主动经营"
- "场景特定性"在 F 节和 D 节各阶段重复

做一次 merge + 交叉引用，估计能砍掉 100-150 行。

#### ☆ 7. Few-shot 补齐

整份 prompt 没有一个**完整的输出示例**。对于这么复杂的风格要求（散文质感 + 第二人称 + 克制 + 不点破），给 2-3 个 few-shot 会比多写 500 字风格描述都有效：
- 阶段一"从更早开始"的苏醒段（C 级，200-300 字）
- 阶段三气泡破碎前的 signal_input_needed 收尾（B 级）
- 冷漠路线女孩介入（A 级）

### 三、对你们引擎需要的改动

1. **`stateSchema` + `update_state` 工具的类型校验**（引擎可能已有）


全是 **one-shot batch 抽取**。编剧唯一的交互点是最后那个 `ResultPreview.tsx`（估计还没接入 UI）——看一眼 AI 抽了啥，点确认。

这种模式的根本问题是：

1. **AI 的假设是不可见的**。它读到"阶段一、阶段二"就抽了两个节点，但它是**基于什么理解**抽的？—— 编剧不知道
2. **AI 的盲点是不可纠正的**。它没问出来的东西，编剧也不会主动想起来补
3. **编剧的隐式规则流失**。MODULE7 里"核心体验建立后可主动推进"这种软规则，one-shot 抽取大概率丢失，因为没有一个字段在明确指向它

### 二、你提的新方向 — 给它一个名字和原语

这本质是在做 **"剧本的可执行规约"(executable spec) 共建**。引擎的职责不是"预设状态机"，是**提供一个对话框架，让 AI 和编剧把"核心循环"显式化到可运行的程度**。

需要的新原语：

#### 1. `EngineUnderstanding` — AI 的结构化阅读理解

```ts
interface EngineUnderstanding {
  /** AI 读出来的核心循环"形状" */
  loopShape: 
    | { kind: 'linear-phases'; phases: PhaseDescription[] }
    | { kind: 'branching-routes'; routes: RouteDescription[] }
    | { kind: 'sandbox'; triggers: TriggerDescription[] }
    | { kind: 'turn-based'; turnRules: string[] }
    | { kind: 'hybrid'; ... };
  
  /** 从 prompt 里读到的状态变量（暂未确认） */
  proposedStateVars: Array<{
    name: string;
    purpose: string;
    evidenceInPrompt: string;  // 哪几行让 AI 这样想的
    confidence: 'high' | 'medium' | 'low';
  }>;
  
  /** 阶段/路径之间的转场条件 */
  proposedTransitions: Array<{
    from: string;
    to: string;
    trigger: string;       // AI 的理解（自然语言）
    evidenceInPrompt: string;
    openQuestions: string[];  // ← 关键：AI 不确定的地方
  }>;
  
  /** AI 识别到的 SOP 硬约束（每阶段必须发生的事） */
  hardCheckpoints: Array<{
    phase: string;
    requirement: string;
    evidenceInPrompt: string;
  }>;
  
  /** AI 自己都没读懂的地方 */
  ambiguities: Array<{
    topic: string;
    aiInterpretation: string;
    alternativeInterpretations: string[];
    needsWriterInput: true;
  }>;
}
```

#### 2. `ReviewDialog` — 编剧逐项审查的结构

```ts
interface ReviewItem {
  id: string;
  kind: 'state-var' | 'transition' | 'checkpoint' | 'ambiguity';
  aiStatement: string;        // "我理解阶段三的退出条件是..."
  aiReasoning: string;         // "我从第 XXX 行读到..."
  writerResponse?:
    | { verdict: 'confirm' }
    | { verdict: 'override'; correction: string }
    | { verdict: 'elaborate'; addition: string }
    | { verdict: 'reject'; reason: string };
}
```

#### 3. `ExecutableSpec` — 已锁定的产物

合并 AI 理解 + 编剧签字 + 溯源。这才是真正进入 IR 的东西：

```ts
interface ExecutableSpec {
  stateSchema: StateSchema;          // 已签字
  segments: PromptSegment[];         // 已签字
  injectionRules: InjectionRule[];   // 已签字
  checkpointTools: ToolDefinition[]; // 已签字
  
  /** 审计信息 —— 后续改 prompt 时回溯用 */
  provenance: {
    fromPromptVersion: string;       // contentHash
    reviewedAt: Date;
    reviewItems: ReviewItem[];       // 完整的对话记录
    unresolvedWarnings: string[];    // 编剧选择"我知道有风险，继续"的地方
  };
}
```

### 三、关键：AI 不只是抽取者，是"提问者"

这点和现在 Architect 最大的区别。比如给 MODULE7 的 prompt，AI 不该直接抽"5 个阶段"，应该输出：

> 我读完了这份 prompt。我的初步理解是你的剧本是**线性阶段 + 分支路径**结构（置信度：高，依据：D 节有明确的阶段预算表）。
>
> 但在做下一步之前，我想先和你对齐 7 个问题：
>
> **Q1 关于状态变量**：你在 H 节列出了 19 个 INTERNAL_STATE 字段。这些我都需要让引擎显式追踪，还是说其中一些（比如 `玩家情绪基调`）你希望 LLM 在上下文里自己判断？这影响字段是做成 state var 还是只做成提示。
>
> **Q2 关于阶段推进**：阶段二到阶段三的转场条件你写的是"玩家在某处感知细节上停留超过2轮，GM 直接用'走了一段之后'"。这个 "2 轮" 是**硬规则**（引擎数，到 2 强制推进）还是**软提示**（LLM 看情况）？
>
> **Q3 关于路径分叉**：冷漠路线和主线的分叉点我没找到明确判定。我推测是"女孩登场后玩家连续 N 轮无互动"，但 N 你没写。你想怎么定？
>
> **Q4 关于第一次开口**：E 节说"后20% 安排触发"。这是 LLM 自行判断的软目标，还是你希望引擎强制（比如在第一章 phase=5 且轮次>阈值时，若 LLM 没触发就 warn）？
>
> **Q5 关于 end_scenario**：目前 prompt 没明说整个剧本的终止条件。第二章阶段四收尾后就算结束吗？还是你计划加第三章？
>
> ...

**每一个问题都带着它的依据（哪几行让 AI 这样想的）和备选答案**。编剧答完这 7 个，engine 就知道该用什么字段、什么条件、什么工具。

### 四、和现有代码的 fit

好消息是**基础设施 90% 都在**：

| 新原语 | 现有代码复用 |
|--------|------|
| `EngineUnderstanding` | `ArchitectResult` 的扩展版，加 evidence / confidence / openQuestions 字段 |
| AI 提问 | 新写一个 `spec-interrogator.ts` agent，用 `generateObject` 但产物是 **questions + hypotheses**，不是直接的 IR |
| 编剧交互 | 新 UI：`SpecReviewPanel.tsx` —— 像 PR review，一条条看，confirm/override/elaborate |
| `ExecutableSpec` | 就是现有 IR（StateSchema + Segments + InjectionRules + Tools），只是多存一个 `provenance` |
| 溯源 | 复用 `contentHash`，编剧改了原文件就触发 re-review |

**真正要新建的只有两个**：
1. `src/core/architect/spec-interrogator.ts` — 一个"提问式" agent
2. `src/ui/architect/SpecReviewPanel.tsx` — 一个交互式审查 UI

#### 引擎侧要保证的

- `stateSchema` 字段**完全由 spec 决定，不硬编码 chapter/phase**。某个剧本可能根本没有"阶段"，只有回合数；另一个可能是 sandbox 无状态
- `ContextAssembler` 的 InjectionRule condition 只依赖 `state.vars[key]`，不知道 key 叫什么——现在应该已经是这样
- `tools` 列表是 per-script 的，checkpoint tool 由 spec 生成，不是引擎内置——`tool-enabler.ts` 已经支持这个方向

### 五、一次对话的样子（MODULE7 示例）

给你看 end-to-end 一次交互大概是什么感觉：

```

Round 2 — 编剧回答

Q3: 3 轮。我把判定条件写清楚了：玩家输入中不含女孩相关关键词（她/娃/孩子/...）

...

Round 3 — AI 整合并再问一轮
"基于你的 Q3 回答，我需要在 state 里加一个 cold_route_counter。

Round 5 — AI 生成最终 ExecutableSpec
```

### 六、一个值得警惕的事

**这个对话 agent 本身就是个 prompt 工程产物**，质量全看问题提得准不准。建议做法：

1. 先手工把 MODULE7 走一遍这个流程（你自己扮演 AI 出 7 个问题，编剧答）
2. 看编剧回答里哪些是"你问得好"，哪些是"你问错了"
3. 用这个做 AI agent 的 few-shot 模板

这比直接让 AI 上来就问靠谱。其实这种 "先人工跑通 → 再 agent 化" 正好也是你们项目过去的节奏（FlowExecutor 都跑通过才敢删）。
