# Interactive Novel Engine v2.5 — 会话持久化 + 可观测性

v2.0 定义了引擎的核心架构（三层架构 + Agentic 工具系统 + Runtime Layer）。v2.5 在此基础上新增两个基础设施能力：

1. **会话持久化**：玩家的游玩记录持久存储，支持断线恢复、跨设备继续、多次游玩管理
2. **可观测性**：LLM 调用质量追踪 + 编剧实时看板

---

## 一、设计哲学

### 会话持久化

**WebSocket 只是传输层，不是状态容器。** 游戏状态持久化在数据库中，WebSocket 断了重连就行。

```
玩家的心智模型：
  "我有几个存档，随时可以继续玩"

不是：
  "我有一个连接，断了就没了"
```

### 可观测性

**编剧需要看到 LLM 在做什么。** 不是事后分析日志，而是在开发/调试过程中实时看到每轮调用的 prompt、token 用量、工具调用、延迟。

```
两个受众，两套工具：

编剧（实时调试）→ DebugPanel 增强（浏览器内）
运维（离线分析）→ Langfuse（独立服务）
```

## 二、数据模型

### 核心实体层级

```
Player（玩家）
 └── Playthrough（游玩记录 — 核心实体）
      ├── script_id    引用哪个剧本（属性，非父级）
      ├── 引擎状态      StateStore + MemoryManager 快照
      └── NarrativeEntry（叙事条目）
```

**Playthrough 是玩家视角的核心实体**，Script 是它的一个属性。一个玩家对同一个剧本可以有多次独立的游玩。

### 数据库 Schema

技术栈：**Drizzle ORM + PostgreSQL**

```typescript
// schema/playthroughs.ts

import { pgTable, text, integer, boolean, jsonb, timestamp } from 'drizzle-orm/pg-core';

export const playthroughs = pgTable('playthroughs', {
  id:               text('id').primaryKey(),              // uuid
  playerId:         text('player_id'),                    // nullable = 匿名玩家
  scriptId:         text('script_id').notNull(),          // 引用的剧本
  title:            text('title'),                        // "游玩 #3" 或自定义
  chapterId:        text('chapter_id').notNull(),
  status:           text('status').notNull().default('idle'),
  turn:             integer('turn').default(0),
  stateVars:        jsonb('state_vars'),                  // StateStore 快照
  memoryEntries:    jsonb('memory_entries'),               // MemoryManager.entries[]
  memorySummaries:  jsonb('memory_summaries'),             // MemoryManager.summaries[]
  inputHint:        text('input_hint'),
  inputType:        text('input_type').default('freetext'),
  choices:          jsonb('choices'),
  preview:          text('preview'),                      // 最近叙事摘要（列表展示用）
  createdAt:        timestamp('created_at').defaultNow(),
  updatedAt:        timestamp('updated_at').defaultNow(),
  archived:         boolean('archived').default(false),    // 软删除
});

export const narrativeEntries = pgTable('narrative_entries', {
  id:             text('id').primaryKey(),
  playthroughId:  text('playthrough_id').notNull()
                    .references(() => playthroughs.id, { onDelete: 'cascade' }),
  role:           text('role').notNull(),                  // generate / receive / system
  content:        text('content').notNull(),
  reasoning:      text('reasoning'),
  toolCalls:      jsonb('tool_calls'),
  finishReason:   text('finish_reason'),
  orderIdx:       integer('order_idx').notNull(),
  createdAt:      timestamp('created_at').defaultNow(),
});
```

### 字段设计说明

| 字段 | 设计决策 |
|------|---------|
| `player_id` | 远程模式从 auth token 取；匿名玩家用 localStorage 随机 ID |
| `preview` | 每次持久化时取最后一条 generate entry 的前 80 字，列表页不需加载全部 entries |
| `archived` | 软删除，玩家可归档旧游玩，数据保留 |
| `updated_at` | 兼作 last_played_at，用于排序 |
| `memory_entries` / `memory_summaries` | JSONB — 引擎内部数据，跟随 session 整体读写，不需独立查询 |
| `narrative_entries` 单独建表 | 会持续增长（几十到几百条），支持分页查询、后续回放/搜索功能 |

## 三、持久化机制

### 持久化时机

**只在状态转换点持久化，不在 streaming 过程中**（太频繁无意义）。

```
                    ┌─ generate 开始 → 存 status='generating'
                    │
  每轮 coreLoop ────┼─ generate 结束 → 存 new narrative_entry + memory 快照
                    │
                    ├─ signal_input_needed → 存 status='waiting-input' + choices
                    │
                    └─ receive 完成 → 存 player entry + state_vars + turn++
```

如果 streaming 中间崩溃，丢失的只是当前轮未完成的生成，恢复时从上一个完整轮次继续。

### 持久化接口

GameSession 在关键节点调用 `SessionPersistence` 接口：

```typescript
interface SessionPersistence {
  /** 更新 playthrough 状态（status, turn, stateVars, choices 等） */
  updatePlaythrough(id: string, patch: Partial<PlaythroughRow>): Promise<void>;

  /** 追加叙事条目 */
  appendNarrativeEntry(playthroughId: string, entry: NarrativeEntryRow): Promise<void>;

  /** 保存 memory 快照（entries + summaries） */
  saveMemorySnapshot(id: string, entries: MemoryEntry[], summaries: string[]): Promise<void>;

  /** 加载完整 playthrough 状态（用于恢复） */
  loadPlaythrough(id: string): Promise<PlaythroughSnapshot | null>;

  /** 加载叙事条目（分页） */
  loadNarrativeEntries(playthroughId: string, limit: number, offset: number): Promise<NarrativeEntryRow[]>;
}
```

### 恢复流程

```typescript
async function restoreSession(playthroughId: string): GameSession {
  const snapshot = await persistence.loadPlaythrough(playthroughId);

  // 重建 MemoryManager
  const memory = new MemoryManager(config);
  memory.restore(snapshot.memoryEntries, snapshot.memorySummaries);

  // 重建 StateStore
  const stateStore = new StateStore(schema);
  stateStore.restore(snapshot.stateVars, snapshot.turn);

  // 重建 GameSession
  const session = new GameSession(emitter);
  session.restore({ memory, stateStore, segments, ... });
  // restore() 跳过 start() 的初始化，直接进入 coreLoop 的对应阶段

  return session;
}
```

GameSession 需要新增 `restore()` 方法：
- 注入已有的 MemoryManager 和 StateStore
- 根据 `status` 决定从 coreLoop 的哪个阶段继续
  - `waiting-input` → 直接进入 Receive Phase
  - `idle` → 等待玩家触发下一轮
  - `generating` → 视为上一轮未完成，回退到 Receive Phase

## 四、WebSocket 重连协议

### 连接流程

```
客户端                                   服务端
  │                                       │
  │ GET /playthroughs?scriptId=xxx        │
  │ ← 游玩列表                            │
  │                                       │
  │ 玩家选择 "继续" 或 "新游玩"             │
  │                                       │
  ├─ 新建 ──→ POST /playthroughs          │ ← 创建 DB 记录
  │           ← { id }                    │
  │           WS /playthroughs/ws/:id     │
  │           ← connected                 │ ← 新建 GameSession
  │                                       │
  ├─ 继续 ──→ WS /playthroughs/ws/:id    │ ← DB 加载 → restoreSession()
  │           ← restored {                │    推送状态快照
  │               entries[],              │
  │               status,                 │
  │               choices,                │
  │               stateVars               │
  │             }                         │
  │                                       │
  │ (断线)                                │ ← GameSession 保持在内存
  │                                       │    （有 TTL，超时后销毁）
  │ WS /playthroughs/ws/:id              │ ← 检测到内存中有活跃 session
  │ ← reconnected { ... }                │    直接恢复，不走 DB
```

### 断线处理策略

```
WebSocket 断开
  │
  ├─ 内存中 GameSession 保留 10 分钟
  │   ├─ 10 分钟内重连 → 直接恢复（零成本）
  │   └─ 超时 → 销毁 GameSession（状态已在 DB）
  │
  └─ 再次连接
      ├─ 内存有 → reconnected（快速）
      └─ 内存无 → 从 DB 恢复（restoreSession）
```

### 客户端 sessionId 存储

localStorage，key 为 `ivn-playthrough-{scriptId}`：

```typescript
interface StoredPlaythrough {
  playthroughId: string;
  timestamp: number;       // 最后活跃时间
}
```

客户端启动时检查：有存储的 playthroughId → 尝试直接 WS 连接恢复；没有 → 显示游玩列表。

### 状态快照推送

恢复连接时，服务端推送完整状态快照：

```typescript
interface StateSnapshot {
  playthroughId: string;
  status: string;
  turn: number;
  stateVars: Record<string, unknown>;
  inputHint: string | null;
  inputType: 'freetext' | 'choice';
  choices: string[] | null;
  entries: NarrativeEntry[];   // 最近 50 条
  totalEntries: number;
  hasMore: boolean;            // 前端可请求更早的 entries
}
```

## 五、API 设计

Playthrough 是一级资源，不嵌套在 Script 下。

```
GET    /api/playthroughs                  -- 当前玩家的所有游玩记录
GET    /api/playthroughs?scriptId=xxx     -- 筛选某剧本
POST   /api/playthroughs                  -- 新建游玩 { scriptId }
GET    /api/playthroughs/:id              -- 游玩详情 + entries（分页）
PATCH  /api/playthroughs/:id              -- 改标题 / 归档
DELETE /api/playthroughs/:id              -- 硬删除

WS     /api/playthroughs/ws/:id           -- 连接到游玩会话
```

### WS 消息类型（新增）

在 v2.0 已有的 WS 消息基础上，新增：

| type | 方向 | 说明 |
|------|------|------|
| `restored` | S→C | 恢复连接，携带 StateSnapshot |
| `reconnected` | S→C | 重连成功（内存中恢复，无需全量快照） |
| `load-more-entries` | C→S | 客户端请求更早的 entries |
| `more-entries` | S→C | 返回更早的 entries 分页 |

## 七、可观测性 — Langfuse 集成（Phase 1）

### 定位

Langfuse 负责 **LLM 调用质量追踪**——每轮 generate 的 prompt/completion/tokens/latency/tool calls。

```
GameSession.generate()
  │
  ├─ Langfuse trace (session 级别)
  │   ├─ generation span (LLM 调用)
  │   │   ├─ input: systemPrompt + messages
  │   │   ├─ output: fullText
  │   │   ├─ usage: inputTokens, outputTokens
  │   │   └─ metadata: model, turn, chapterId
  │   │
  │   ├─ span: tool:signal_input_needed
  │   ├─ span: tool:update_state
  │   └─ span: tool:roll_dice
  │
  └─ 每轮结束 → trace.end()
```

### 接入方式

使用 Langfuse JS SDK（服务端）：

```typescript
import { Langfuse } from 'langfuse';

// 初始化（server 启动时）
const langfuse = new Langfuse({
  secretKey: process.env.LANGFUSE_SECRET_KEY,
  publicKey: process.env.LANGFUSE_PUBLIC_KEY,
  baseUrl: process.env.LANGFUSE_HOST,   // 自部署地址，或 https://cloud.langfuse.com
});

// GameSession.generate() 中
const trace = langfuse.trace({
  name: 'game-generate',
  sessionId: playthroughId,
  userId: playerId,
  metadata: { turn, chapterId, scriptId },
});

const generation = trace.generation({
  name: 'llm-call',
  model: this.config.model,
  input: { system: systemPrompt, messages },
  modelParameters: { maxOutputTokens },
});

// ... streamText 完成后 ...

generation.end({
  output: fullText,
  usage: {
    input: tokenBreakdown.total - outputTokens,
    output: outputTokens,
  },
});

// tool calls 作为子 span
for (const tc of toolCalls) {
  trace.span({
    name: `tool:${tc.name}`,
    input: tc.args,
    output: tc.result,
  });
}
```

### 环境配置

```env
# .env (server)
LANGFUSE_SECRET_KEY=sk-lf-...
LANGFUSE_PUBLIC_KEY=pk-lf-...
LANGFUSE_HOST=https://cloud.langfuse.com   # 或自部署地址
```

开发阶段使用 Langfuse Cloud 免费额度（50k observations/月），正式上线可切换为自部署（Docker Compose）。

### Langfuse 中的数据视图

```

│   └── 游玩 abc-123
│       ├── Turn 1: generation (820 input tok, 340 output tok, 2.1s)

    └── Per-session / per-model cost
```

## 八、可观测性 — DebugPanel 增强（Phase 2）

### 定位

DebugPanel 是**编剧在浏览器中的实时看板**，不依赖外部服务。数据来源是 GameStore 中已有的信息 + 新增的累计统计。

### 新增 SessionStats

```typescript
interface SessionStats {
  totalInputTokens: number;
  totalOutputTokens: number;
  turnStats: TurnStat[];           // 每轮统计
  finishReasons: Record<string, number>;
  toolCallCounts: Record<string, number>;
}

interface TurnStat {
  turn: number;
  inputTokens: number;
  outputTokens: number;
  latencyMs: number;               // 首 token 到最后 token
  toolCalls: string[];             // 本轮调用的工具名
  finishReason: string;
}
```

GameStore 新增 `sessionStats` 字段，在每轮 `finalizeStreamingEntry` 时累计更新。

### UI 设计

在 DebugPanel 中新增"统计"标签页：

```
┌─ 状态 ─ 记忆 ─ [统计] ─────────────────────┐

│  本次会话                                    │

│                                             │
```

### 数据收集点

| 数据 | 来源 | 时机 |
|------|------|------|
| Token 用量 | LLM generate result | finalizeStreamingEntry |
| 延迟 | beginStreamingEntry → finalizeStreamingEntry 的时间差 | finalizeStreamingEntry |
| Tool calls | onToolCall 回调 | 实时 |
| Finish reason | generate result | finalizeStreamingEntry |
