Trace 时间线排序问题分析

核心问题：AI SDK 的 onStepFinish 在 step 的所有 tool 结果返回后才触发。
对 signal_input_needed，tool Promise 挂起等玩家回复（可能几分钟），
所以 onStepFinish 的触发时间 = 玩家回复时间 ≠ LLM 实际生成时间。

问题场景：Step-2 的时间戳被"拖晚"

实际发生顺序（真实时间）
T+0s: LLM 开始 step-2，生成叙事文本
T+16s: LLM 输出完叙事 + 调用 signal_input_needed
T+16s: tool:signal_input_needed span START（Langfuse 能看到）
T+16s ~ T+5min: 等待玩家输入...
T+5min: 玩家回复
T+5min: tool:signal_input_needed span END
T+5min: onStepFinish 触发 → recordStep
修复前：recordStep 用 new Date() → 时间戳 = T+5min ❌
修复后：recordStep 用 step.response.timestamp → 时间戳 ≈ T+0s ✅
Langfuse UI 中的显示效果对比
修复前：
step-0-tool ───── T+0s
step-1-tool ───── T+2s
signal_input ──── T+16s ~ T+5min
player_input ──── T+5min
step-2-narrative ─ T+5min+1ms   ← 看起来在 player_input 之后！
修复后：
step-0-tool ───── T+0s
step-1-tool ───── T+2s
step-2-narrative ─ T+2s     ← 正确位置：在 signal 之前
signal_input ──── T+16s ~ T+5min
player_input ──── T+5min
仍然存在的显示问题
step.response.timestamp 是 "LLM 响应开始的时间"，不是"结束的时间"。
所以 step-2 的时间戳 ≈ step-1 结束后 → 看起来 step-2 和 step-1 几乎同时，
但实际上 step-2 的生成过程持续了 ~16 秒。

不影响排序正确性，但 Langfuse UI 里 step-2 的"持续时间"显示为 0ms
（因为 startTime = endTime = response.timestamp）。
如果需要精确的 duration，需要额外捕获 step 的"最后一个 chunk 到达"时间。
