# Interactive Novel Engine v2.0 — Architecture Design

## Design Philosophy

**编剧只写文字。引擎负责理解。LLM 驱动一切。**

```
编剧的工作：写 Prompt 文档 + 世界观资料 + 场景设定 — 全部是纯文字
引擎的工作：Agent 从文字中提取结构 → 可视化流程图让编剧确认/调整 → 执行
```

类比编译器：
```
高级语言（编剧的文字）→ 编译（Agent 提取）→ IR（内部数据结构）→ 执行（引擎运行时）
                                ↕
                        可视化 IDE（流程图 + 节点编辑器）
```

编剧不需要学 YAML、模板语法、或任何技术记法。他们的 15,000 字 GM Prompt **已经包含了一切**——阶段地图、NPC 行为规则、节奏预算、收束条件——只是用自然语言写的。引擎通过 Agent 读懂这些文字，自动生成可执行的结构。

### 核心循环的第一性原理

从第一性原理看，交互式叙事的原子操作**只有两个**：

```
① Generate（引擎产出内容）— LLM 以 Agentic 模式运行，自主调用工具
② Receive（引擎接收输入）— 等待玩家（或自动代理）的输入
```

整个游戏就是这两个操作的序列：

```
G → G → R → G → R → G → G → R → ...
│   │       │       │   │
序章  行走   玩家回应  GM回应  两段独白  玩家回应
(auto)(auto)
```

**所有其他东西——记忆、状态、压缩、Prompt 组装、token 预算——都是 Generate 内部的实现细节。** 它们服务于"如何更好地执行 Generate"，不是独立的顶层步骤。

路由（决定下一步是 G 还是 R）不是第三个操作——**LLM 通过 `signal_input_needed` 工具自主决定何时暂停等待玩家输入**。引擎不做路由判断。

## 一、三层架构

```
┌────────────────────────────────────────────────────────────────┐
│  Authoring Layer（编剧接触的层）                                │
│                                                                 │
│  输入：纯文字文档（Prompt、世界观、场景设定）                     │
│  输出：可视化流程图 + 结构确认界面                                │
│  工具：Architect Agent（文字→结构转译）+ Visual Flow Editor      │
├────────────────────────────────────────────────────────────────┤
│  Data Layer — IR（中间表示，对编剧透明）                         │
│                                                                 │
│  Prompt Segments（拆分后的 Prompt 片段）                         │
│  State Schema（状态变量定义）                                    │
│  Memory Config（记忆策略配置）                                   │
│  FlowGraph（可视化组织参考图，非运行时控制流）                    │
│  Enabled Tools（编剧启用的工具列表）                              │
├────────────────────────────────────────────────────────────────┤
│  Runtime Layer（执行引擎）                                       │
│                                                                 │
│  CoreLoop (Generate + Receive) + LLM Client + ToolExecutor     │
│  ContextAssembler + MemoryManager + StateStore + UI Renderer    │
└────────────────────────────────────────────────────────────────┘
```

## 二、Authoring Layer — 编剧的工作流

### 编剧做什么

编剧的全部工作就是**写文档**，和他们现在做的事情完全一样：

```
module-7/
  docs/
    GM提示词_序章第一章.md        ← 就是现有的 15,000 字文档
    GM提示词_第二章_共鸣池.md
    PC提示词.md
    世界观时间线.md
    共鸣池场景设定.md
    角色设定_女孩.md
```

没有 `flow.yaml`。没有 `state.yaml`。没有模板语法。纯文字。

### Architect Agent 做什么

编剧将文档导入引擎后，Architect Agent（一个 LLM 驱动的分析流程）阅读所有文档，自动提取：

```
输入：编剧的全部文档
  │
  ▼
Architect Agent 分析提取：
  │
  ├─ 1. State Variables（状态变量）
  │     "girl_language_level: 0-3, GM会更新"
  │     "explored_areas: 列表, 追踪已探索区域"
  │     从 GM Prompt 的 INTERNAL_STATE 定义中提取
  │
  ├─ 2. Prompt Segments（Prompt 拆分）
  │     "这段是通用写作规范 → 可复用"
  │     "这段是第一章阶段地图 → 章节特定"
  │     "这段是共鸣池设定 → 按需注入"
  │     从文档结构和内容语义中拆分
  │
  ├─ 3. Memory Hints（记忆策略建议）
  │     "跨章时需要保留：女孩说过的话、玩家选择、信任等级"
  │     从 GM Prompt 中关于"前章记录继承"的描述提取
  │
  ├─ 4. Injection Rules（动态注入规则）
  │     "当 current_area == 'resonance-pool' 时，注入共鸣池设定"
  │     "当 girl_language_level >= 2 时，使用不同的行为描述"
  │     从 GM Prompt 中的条件性描述提取
  │
  ├─ 5. Tool Enable List（工具启用列表）
  │     从编剧文字中提取哪些可选工具应该启用
  │     根据状态变量定义自动生成 update_state 的参数 schema
  │
  └─ 6. FlowGraph（可视化组织参考图）
        "序章：玩家苏醒 → 行走 → 遇到女孩 → ..."
        "第二章：进入共鸣池 → 涅瓦 → 卡伦 → ..."
        从 GM Prompt 的阶段地图、收束协议中提取
        ⚠️ 此图仅供编剧在可视化编辑器中确认结构，不作为运行时控制流
```

### Visual Flow Editor — 编剧确认/调整的界面

Agent 提取完成后，引擎呈现一个**可视化流程图**。

**FlowGraph 的定位：可视化组织参考图。** 它帮助编剧直观地看到 Agent 从文字中提取的叙事结构是否正确。但运行时，LLM 不会被 FlowGraph "驱动"——LLM 根据 Prompt 中的阶段地图和收束协议自主决定叙事推进，这正是编剧写 Prompt 的方式。

编剧可以：
- **确认**：Agent 提取的结构是否正确
- **拖拽**：调整场景顺序、添加/删除节点
- **编辑**：点击节点查看/修改对应的 Prompt 片段
- **修改状态变量**：重命名、添加、删除
- **调整注入规则**：哪个 Prompt 片段在什么条件下注入

**所有编辑都是可视化操作，不需要写代码或 YAML。**

### 双向工作流

```
编剧修改了文档 → 重新运行 Architect Agent → 流程图更新 → 编剧确认
编剧在流程图上调整 → IR 更新 → 引擎按新结构运行
编剧在流程图上发现问题 → 回到文档修改 → 重新提取 → 循环
```

### 多轮精化

Architect Agent 不是一次性提取。它可以多轮交互：

```
Agent: "我从您的第一章 GM Prompt 中提取了 5 个阶段：苏醒、行走、邂逅、陪伴、入睡。
        流程图如下 [显示可视化]。请确认是否正确？"
编剧: "缺少了'坠落'阶段，在邂逅和陪伴之间"
Agent: [更新流程图，添加节点]
编剧: "对了，但坠落阶段有一个分支——玩家可以选择伸手或不伸手"
Agent: [添加分支]
```

## 三、Data Layer — IR（中间表示）

IR 是 Agent 提取的结构化数据。编剧不直接编写它，但可以通过可视化编辑器间接修改它。高级用户（程序员）也可以直接编辑 IR。

### Prompt Segments

```typescript
interface PromptSegment {
  id: string;
  label: string;                    // "GM基础规范", "第一章阶段地图", "共鸣池设定"
  content: string;                  // 原文内容
  contentHash: string;              // 内容哈希，用于检测变化
  type: 'content' | 'logic';       // content=纯内容段, logic=控制逻辑段
  sourceDoc: string;                // 来源文档路径
  sourceRange?: [number, number];   // 来源文档中的行号范围
  role: 'system' | 'context';      // system=始终注入, context=按需注入
  priority: number;                 // 组装时的优先级
  injectionRule?: InjectionRule;    // 何时注入
  tokenCount: number;
}

interface InjectionRule {
  description: string;              // 自然语言描述，编剧可见
  condition: string;                // 编译后的条件表达式
}
```

**Segment 类型说明：**
- `content` — 纯内容段（世界观、场景、角色设定）
- `logic` — 控制逻辑段（定义激活规则、流程条件、工具使用策略）

**Segment 激活状态管理：**
- 存档保存当前激活的 segment ID 列表 + 剧本版本号，不存内容快照
- 剧本更新后，通过 contentHash 比对检测哪些 segment 有变化
- **logic 类型 segment 变化** → 触发重新计算激活列表
- **content 类型 segment 变化** → 只更新内容，不重算激活列表
- 首次检测到变化时提示编剧；剧本版本信息中保留变化列表供重复查看
- `inject_context` 手动拉取的文档为一次性注入，只影响当前轮

**Prompt Segment 的来源**：Architect Agent 将编剧的大文档智能拆分为片段。例如：

```
GM提示词_第二章.md (15000字)
  → Segment: "通用写作规范" (2000字, system, always)
  → Segment: "第二章阶段地图" (3000字, system, always when chapter=2)
  → Segment: "涅瓦区域细节" (800字, context, when current_area=neva)
  → Segment: "卡伦区域细节" (600字, context, when current_area=caren)
  → Segment: "自检清单" (500字, system, always)
  → ...
```

编剧在可视化编辑器中可以看到拆分结果，调整边界和注入规则。

### State Schema

```typescript
interface StateSchema {
  variables: StateVariable[];
}

interface StateVariable {
  name: string;
  type: 'number' | 'string' | 'boolean' | 'array' | 'object';
  initial: unknown;
  description: string;            // 自然语言描述
  range?: { min?: number; max?: number };  // 数值范围（如 language_level 0-3）
}
```

注意：不再有 `updatedBy` 字段。**所有状态更新都通过 LLM 的 `update_state` tool call**，这是设计讨论中确定的核心原则（见 Turn 15）。

### Memory Config

```typescript
interface MemoryConfig {
  contextBudget: number;
  compressionThreshold: number;
  recencyWindow: number;
  compressionHints?: string;        // 自然语言的压缩指导
  crossChapterInheritance?: CrossChapterConfig;
}

interface CrossChapterConfig {
  inherit: string[];                // 编剧显式要求继承的字段
  exclude: string[];                // 编剧显式要求不继承的字段
  // 未声明的字段由 Architect Agent 自动推断，结果展示给编剧确认
}
```

**跨章继承机制：**
- 章节切换时执行继承操作，同时**保存继承快照**到新章节状态（`inheritedFrom: { chapter, fields, values }`）
- 继承规则三层 fallback：
  1. **编剧显式声明**（最高优先级）：inherit / exclude 列表
  2. **Architect Agent 自动推断**（兜底）：分析两章 StateSchema，同名字段默认继承
  3. **GM 运行时补充**（可选）：上一章结束时 GM 通过 tool call 额外标记
- Architect Agent 推断后须在可视化界面中展示给编剧确认，编剧可排除不需要的字段
- 编剧需被提醒：未声明的字段会被自动推断继承，如不需要须明令禁止

### FlowGraph（可视化组织参考图）

```typescript
interface FlowGraph {
  nodes: FlowNode[];
  edges: FlowEdge[];
}

interface FlowNode {
  id: string;
  label: string;              // 编剧可见的名称，如"序章·苏醒"
  description?: string;       // 该节点对应的叙事阶段简述
  promptSegments: string[];   // 关联的 Prompt 片段 ID 列表
}

interface FlowEdge {
  from: string;
  to: string;
  label?: string;             // 边上显示的标签
}
```

**关键设计原则：FlowGraph 不是运行时控制流图。**

FlowGraph 的唯一用途是：
1. **Architect Agent 提取**：从编剧文字中提取叙事结构，生成可视化供编剧确认
2. **Visual Flow Editor**：编剧在编辑器中查看和调整叙事的组织结构
3. **Prompt Segment 关联**：通过节点关联相关的 Prompt 片段，帮助组织 IR

运行时，LLM 根据 Prompt 中的阶段地图和收束协议自主推进叙事。引擎不遍历 FlowGraph 的节点和边来决定"下一步做什么"。这与编剧原始的工作方式一致——编剧在 GM Prompt 中用自然语言描述了所有流程控制（"当核心体验建立后进入下一阶段"、"当玩家探索完所有区域后收束"），LLM 读懂这些指令后自主执行。

## 四、Architect Agent — 从文字到结构

### Agent 的工作流程

```
Step 1: 文档分类
  输入：编剧上传的所有文档
  输出：文档角色标注
    "GM提示词_序章第一章.md" → type: gm_prompt, chapter: 1
    "世界观时间线.md" → type: world_data
    "共鸣池场景设定.md" → type: location_data

Step 2: 状态变量提取
  输入：GM Prompt 中的 INTERNAL_STATE 定义
  输出：StateSchema
  方法：GM Prompt 通常会明确列出需要追踪的变量

Step 3: Prompt 拆分
  输入：完整的 GM Prompt 文档
  输出：PromptSegment[]（含 type 标记 + contentHash）
  方法：按语义和功能拆分——通用规范、章节特定、区域特定、条件性内容

Step 4: 注入规则生成
  输入：GM Prompt 中关于"何时使用何种信息"的描述
  输出：InjectionRule[]
  方法：从条件性描述中提取

Step 5: 工具启用列表 + update_state schema 自动生成
  输入：编剧文字中关于工具使用的描述 + 状态变量定义
  输出：enabledTools[] + update_state 参数 schema

Step 6: 流程结构提取（可视化参考图）
  输入：GM Prompt 中的阶段地图、收束协议
  输出：FlowGraph（仅供可视化确认）
  方法：从"阶段A → 条件 → 阶段B"的描述中提取节点和边

Step 7: 记忆策略生成
  输入：GM Prompt 中关于记忆/继承的描述
  输出：MemoryConfig
  方法：从跨章继承说明中提取需要保留的信息
```

## 五、Agentic 工具系统

### 设计理念

GM（LLM）是一个**拥有工具的 Agent**。状态更新、记忆操作、节奏控制等都通过 LLM tool call 完成。引擎提供工具，编剧通过 prompt 指导 GM 何时使用。

**LLM 自主控制一切：**
- **状态更新**：LLM 自律调用 `update_state` 维护游戏状态
- **节奏控制**：LLM 根据 prompt 自主判断何时调用 `signal_input_needed` 暂停
- **记忆管理**：LLM 自主决定何时 pin_memory、query_memory
- **流程推进**：LLM 根据编剧在 prompt 中写的阶段地图自主推进

引擎只做一件事：**执行 LLM 的 tool call 请求**。

### 工具清单

**必选工具（引擎自动注入，编剧不需要关心）：**

| 工具 | 参数 | 功能 |
|------|------|------|
| `update_state(patch)` | `patch: Record<string, unknown>` | 部分更新 ScriptState 中的变量 |
| `signal_input_needed(prompt_hint?)` | `prompt_hint?: string` | 告诉引擎"该等玩家了"，GM 自主控制停顿时机 |

**可选工具（编剧在 prompt 中决定是否启用）：**

| 工具 | 参数 | 功能 |
|------|------|------|
| `read_state(keys?)` | `keys?: string[]` | 读取当前 ScriptState，不传 keys 返回全量 |
| `query_changelog(filter)` | `filter: { key?, turnRange?, timeRange? }` | CRUD 查询状态变更历史 |
| `pin_memory(content, tags?)` | `content: string, tags?: string[]` | 标记重要记忆，压缩时优先保留 |
| `query_memory(query)` | `query: string` | 语义搜索历史记忆 |
| `inject_context(doc_id)` | `doc_id: string` | 一次性拉取世界知识文档到当前上下文（只影响当前轮） |
| `list_context()` | 无 | 列出所有可用的世界知识文档 ID 和简介 |
| `set_mood(mood)` | `mood: string` | 设置场景氛围标签，UI 据此切换视觉/音频风格 |
| `show_image(asset_id)` | `asset_id: string` | 触发 UI 显示图片/CG |

一共 10 个工具，核心 8 个，渲染域 2 个按项目需要。

### 编剧如何引用工具

**两层机制：**

**第一层：引擎自动注入** — 引擎在组装 prompt 时，自动把启用工具的 JSON Schema 定义附加到 system prompt。编剧不需要知道工具的技术定义。

**第二层：编剧用自然语言指导 GM 何时使用** — 编剧在 prompt 中用自然语言描述使用时机：

```markdown
## 工具使用指南

### 状态更新
- 当女孩的语言能力发生变化时（从无声到单词、从单词到短句），
  立即更新 girl_language_level
- 当玩家进入一个新区域时，将区域名加入 explored_areas

### 重要记忆
- 当发生以下情况时，将其标记为重要记忆：
  - 玩家对女孩做出承诺
  - 玩家发现关键的世界观线索

### 查询记忆
- 当你需要引用之前的对话细节时，先查询确认，不要凭印象编造

### 场景切换
- 每一段叙事结束后，如果需要玩家回应，告知引擎等待输入
- 如果当前阶段是自动过场（如"觉醒"阶段），连续生成直到需要玩家参与的节点再停下
```

编剧写的完全是自然语言，不需要知道 JSON Schema。

**Architect Agent 的角色**：从编剧文字中提取工具启用列表 + 根据状态变量定义自动生成 `update_state` 的参数 schema。

**编辑器增强（后续）**：编剧在编辑器中通过"/"菜单选择工具引用（类似 Notion slash command），表面渲染为可读标签（如"查询历史记忆"），底层存储为机器标识（`query_memory`）。

## 六、Runtime Layer — 执行引擎

### 核心循环

运行时的核心循环只有两个操作：**Generate** 和 **Receive**。

```
┌──────────────────────────────────────────────────┐
│                  Core Loop                        │
│                                                   │
│  while (session.active) {                         │
│                                                   │
│    ┌─── Generate ─────────────────────────────┐  │
│    │  1. ContextAssembler 组装 Prompt          │  │
│    │     - 按 injectionRule 筛选 segments      │  │
│    │     - 注入 state 快照                     │  │
│    │     - 注入 memory summaries + 近期对话    │  │
│    │     - 注入工具定义                         │  │
│    │     - 按 token 预算裁剪                   │  │
│    │                                           │  │
│    │  2. LLM Agentic Loop (streamText)         │  │
│    │     - text chunk → 流式渲染到 UI          │  │
│    │     - tool_call(update_state) → 执行      │  │
│    │     - tool_call(query_memory) → 搜索返回  │  │
│    │     - tool_call(pin_memory) → 执行        │  │
│    │     - tool_call(signal_input_needed)       │  │
│    │       → 标记需要玩家输入                   │  │
│    │     - ... 其他工具调用 ...                 │  │
│    │     LLM 可能连续多步（maxSteps），         │  │
│    │     直到自然结束或调用 signal_input_needed │  │
│    │                                           │  │
│    │  3. Memory.append(GM 输出)                │  │
│    │  4. 检查是否需要压缩                       │  │
│    └───────────────────────────────────────────┘  │
│                                                   │
│    if (inputNeeded) {                             │
│                                                   │
│    ┌─── Receive ──────────────────────────────┐  │
│    │  1. 通知 UI 等待输入                      │  │
│    │     （可显示 prompt_hint 引导文字）         │  │
│    │  2. 等待玩家输入                          │  │
│    │  3. Memory.append(玩家输入)               │  │
│    └───────────────────────────────────────────┘  │
│                                                   │
│    }                                              │
│  }                                                │
└──────────────────────────────────────────────────┘
```

**为什么没有"路由"步骤？**

在 Turn 10 的设计讨论中确认：路由不是第三个操作。LLM 根据编剧在 Prompt 中写的阶段地图自主决定叙事推进方向。`signal_input_needed` 是 LLM 自主决定"该停了"的工具——这让 GM 可以在自动过场时连续生成多段叙事而不暂停，也可以在需要玩家参与时随时停下。

**与编剧原始工作方式的一致性：**

编剧给 Claude/LLM 所有的文档，以 prompt.txt 中的内容作为提示词，就可以直接玩起来了。v2.0 引擎做的事情就是把这个过程工程化——自动组装上下文、管理记忆压缩、提供结构化的工具调用——但核心循环的逻辑和"直接跟 LLM 对话"完全一致。

```
┌─────────────────── UI Layer ───────────────────┐
│  NarrativeView  │  InputPanel  │  DebugPanel   │
│  (流式渲染)      │  (文本/选项) │  (状态/记忆)   │
└───────▲──────────────┬──────────────▲──────────┘
        │        useGameStore         │
        │         (Zustand)           │

│                                                  │
│  Generate:                                       │
│    1. ContextAssembler 根据注入规则组装 Prompt   │
│    2. 注入工具定义（必选 + 编剧启用的可选工具）  │
│    3. LLM Agentic Loop:                          │
│       - 流式文本 → UI 渲染                       │
│       - tool call → ToolExecutor 执行 → 返回结果 │

│                                                  │
└──────────────────────────────────────────────────┘
│  ContextAssembler  │  ToolExecutor  │  LLM Client  │  Memory  │  State  │
└──────────────────────────────────────────────────────────────────────────┘
```

### ContextAssembler — Prompt 组装

ContextAssembler 是运行时中最关键的组件。它根据当前状态，从 Prompt Segments 中选择性组装 LLM 的输入：

```
每次 Generate 时:

1. 收集所有 Prompt Segments
2. 对每个 segment 检查 injectionRule:
   - 无规则 → 始终注入
   - 有规则 → 根据当前 ScriptState 求值条件，true 才注入
3. 按 priority 排序
4. 加入 state 快照（序列化为 YAML）
5. 加入 memory summaries（压缩摘要）
6. 加入 recent history（近期原文）
7. 按 token 预算裁剪:
   - system segments（不可裁）> memory summary > recent history > context segments
```

### ToolExecutor — 工具注册与执行

ToolExecutor 负责注册工具定义（JSON Schema）并在 LLM 发出 tool call 时执行：

```typescript
interface ToolHandler {
  description: string;
  parameters: ZodSchema;        // Zod schema，引擎自动转为 JSON Schema
  execute: (args: unknown) => unknown;
  required: boolean;            // true = 必选工具，false = 可选
}
```

引擎启动时注册所有工具：
- 必选工具（`update_state`、`signal_input_needed`）始终注册
- 可选工具根据 IR 中的 `enabledTools` 列表注册

工具的执行是纯本地操作（< 1ms），除 `query_memory` 约 10-50ms。

基于 Vercel AI SDK 的 `streamText` + `tools` + `maxSteps` 实现。AI SDK v6 注意事项：
- `tool()` 的参数名为 `inputSchema`（不是 `parameters`）
- Zod v4 的 schema 需要用 `zodSchema()` 包装

### Memory Manager

```
entries: [E0, E1, E2, ..., E_n]     全量记录
summaries: [S0, S1, ...]             分层摘要
watermark: k                          压缩水位线

压缩时：
  保留最近 recencyWindow 条原文
  其余 → LLM 生成摘要（可用编剧提供的 hintPrompt 指导）
  摘要累计过长 → 合并旧摘要
  → O(log N) 增长

压缩器可从 Changelog 拉取状态变更记录作为参考输入，
但 Changelog 本身永不被修改或删除。
```

### State Store

```
ScriptState: KV 存储（当前快照）
  get(key) / set(key, value) / update(patch)
  serialize() → YAML（注入 Prompt 用）
  export() / import() — 跨章/存档

Changelog: 独立存储（变更历史）
  append({ turn, key, prev, next })
  query(filter: { key?, turnRange?, timeRange? })
  永不压缩/删除，保持原始数据纯净
  容量不设上限
  查询是传统 CRUD，不走语义搜索
  记忆压缩时可被 compressor 拉取作为参考输入

状态更新来源：
  GM 通过 tool call 调用 update_state(patch) — 唯一来源
```

## 七、引擎最小状态集

恢复一个暂停的游戏需要以下四类状态：

```
         │  引擎拥有（引擎管理）          │  剧本拥有（编剧定义，引擎存储）
─────────┼───────────────────────────────┼─────────────────────────
持久化    │  ① Progress    ③ Memory      │  ② Script State
（存档必须）│                               │
─────────┼───────────────────────────────┼─────────────────────────
临时      │  ④ Runtime                    │  （无）
（丢了能重建）│                               │
```

### ① Progress — 当前进度

```typescript
interface ProgressState {
  totalTurns: number;
  inputNeeded: boolean;           // 当前是否在等待玩家输入
  activeSegmentIds: string[];     // 当前激活的 segment ID 列表
  scriptVersion: string;          // 剧本版本号
}
```

### ② Script State — 剧本内部状态

```typescript
interface ScriptState {
  vars: Record<string, unknown>;
}
// 例如 MODULE_7：{ current_stage: "journey", girl_language_level: 0, ... }
```

更新来源：LLM 通过 `update_state` tool call。

### ③ Memory — 记忆模块状态

```typescript
interface MemoryState {
  entries: MemoryEntry[];
  summaries: string[];
  watermark: number;
  tokenCounts: number[];
  inheritedSummary?: string;
}
```

### ④ Runtime — 临时执行状态

```typescript
interface RuntimeState {
  status: 'idle' | 'generating' | 'waiting-input' | 'compressing' | 'error';
  streamBuffer?: string;
  lastTokenUsage?: { system: number; history: number; summary: number; output: number; total: number; budget: number };
  error?: { message: string; recoverable: boolean };
}
```

存档 = 持久化 ①②③ + Changelog。恢复时 ④ 重建。

## 八、完整工作流示例

### Step 1: 编剧上传文档

编剧将已有的 MODULE_7 文档导入引擎：
- GM提示词_序章第一章_v2_3.md
- GM提示词_第二章_共鸣池_v1_9.md
- PC提示词.md
- 世界观时间线.md
- 共鸣池场景设定.md

### Step 2: Architect Agent 分析

Agent 阅读所有文档，输出：

**提取的状态变量：**
```
✅ current_stage: string      — "从GM Prompt的INTERNAL_STATE中识别"
✅ girl_language_level: 0-3   — "从语言发展阶段描述中识别"
✅ player_behavior_tendency   — "从玩家画像分析描述中识别"
✅ explored_areas: array      — "从共鸣池分区探索追踪中识别"
✅ key_memory_points: array   — "从关键记忆锚点描述中识别"
...
```

**提取的 Prompt 片段：**
```
📄 GM通用规范 (所有章节共用, 2100 tokens, type: logic)
📄 自检清单 (所有章节共用, 800 tokens, type: logic)
📄 第一章阶段地图 (chapter=1, 3200 tokens, type: logic)
📄 第一章NPC行为指引 (chapter=1, 1800 tokens, type: content)
📄 第二章阶段地图 (chapter=2, 2900 tokens, type: logic)
📄 共鸣池·涅瓦描述 (current_area=neva, 600 tokens, type: content)
📄 共鸣池·卡伦描述 (current_area=caren, 500 tokens, type: content)
📄 世界观时间线 (按需, 1200 tokens, type: content)
```

**提取的可视化流程图：**
```
[序章·苏醒] → [行走·荒漠] → [邂逅·女孩] → [第一章主循环] → [记忆压缩·章节过渡] → [第二章·共鸣池]
```

### Step 3: 编剧在可视化编辑器中确认

编剧看到流程图和提取结果，进行调整：
- "坠落阶段缺少了" → 在图上添加节点
- "共鸣池的探索顺序不应该是固定的" → 调整结构
- "涅瓦的描述应该拆成两段" → 点击片段，调整拆分

### Step 4: 确认后运行

编剧点击"开始"，引擎进入核心循环。

**Generate（第一轮 — 自动过场）：**

1. ContextAssembler 组装 prompt（~29k tokens）：
   - P1 — System Prompt: GM 核心规范 + NPC行为 + 写作原则 + 自检清单（~25k）
   - P2 — State 快照：`current_stage: awakening, turn_count: 0, ...`（~1k）
   - P3 — 世界知识：时间线摘要（~3k）
   - 无历史消息（第一轮）

2. LLM Agentic Loop：
   - GM 生成觉醒阶段叙事（~2000字）→ 流式渲染到 UI
   - `tool_call(update_state, {current_stage: "journey", turn_count: 1, ...})` → 执行
   - GM 继续生成行走阶段叙事（自动过场，不暂停）→ 流式渲染
   - `tool_call(update_state, {current_stage: "encounter", ...})` → 执行
   - GM 生成邂逅场景，判断需要玩家回应
   - `tool_call(signal_input_needed, {prompt_hint: "你可以用【行動】【対話】【思考】格式回应"})` → 标记

3. Memory.append(GM 全部输出)
4. 压缩检查：未达阈值

**Receive：**

1. UI 显示 InputPanel，提示文字为 prompt_hint
2. 玩家输入：`【行動】慢慢地坐起身来...【思考】这是什么地方...`
3. Memory.append(玩家输入)

**Generate（第二轮）→ Receive → Generate → ... 循环继续**

### 压缩触发

假设玩到第 15 轮，Memory 累计 ~55k tokens，超过阈值：
1. 保留最近 recencyWindow 条原文
2. 其余 → LLM 生成摘要（compressor 可从 Changelog 拉取状态变更作为参考）
3. 下次组装时注入摘要 + 近期原文

### 跨章继承

第一章结束进入第二章：
1. 压缩第一章所有剩余原文为最终摘要
2. 按三层 fallback 规则合并状态（编剧声明 > Agent 推断 > GM 补充）
3. 保存继承快照
4. 新章节 Memory 起始状态：entries 为空，inheritedSummary = 第一章最终摘要
5. 切换 Prompt Segments 为第二章的内容

## 九、与 v1 的对比

| 维度 | v1 (当前仓库) | v2 |
|------|-------------|-----|
| **编剧写什么** | JSON 表单 | 纯文字文档（他们已经在写的） |
| **结构化工作** | 编剧手动完成 | Agent 自动提取 + 编剧可视化确认 |
| **游戏循环** | 引擎硬编码 9 步 | Generate + Receive（LLM 驱动） |
| **流程控制** | 引擎遍历 FlowGraph | LLM 根据 Prompt 自主决定 |
| **状态更新** | 解析 fenced block | LLM tool call（结构化、有 Schema 校验） |
| **暂停时机** | 引擎按 node type 决定 | LLM 自主调用 signal_input_needed |
| **Prompt 管理** | 固定 JSON 字段 | 智能拆分为片段 + 按需注入 |
| **记忆** | 单层关键词匹配 | 分层压缩 + O(log N) |
| **Token 管理** | 无 | 自动预算分配 |
| **编剧门槛** | 需要理解 JSON/GOAP | 只需要会写文字 + 看懂流程图 |

### 本质差异

```
v1:  编剧 → [手动转译为JSON] → 引擎硬编码流程执行
v2:  编剧 → [Agent 自动转译] → [编剧可视化确认] → LLM 驱动的 Agentic 执行
```

## 十、关键技术挑战

### 1. Architect Agent 的提取质量

**问题**：Agent 能否准确从 15,000 字的自然语言中提取出正确的结构？

**策略**：
- 分步提取（先提变量、再拆 Prompt、再提流程），每步用专门的 prompt
- 多轮确认——Agent 提取后，编剧在可视化界面上校正
- 编剧的文档写得越结构化，提取越准确
- 对于模糊的部分，Agent 标记"不确定"让编剧手动确认

### 2. Prompt 拆分的粒度

**问题**：把一个 15,000 字文档拆成多少个片段？太粗 = 无法按需注入；太细 = 上下文断裂。

**策略**：
- 以文档自身的 Markdown 标题结构为初始拆分点
- Agent 根据语义相关性建议合并或再拆分
- 编剧在可视化界面中可以调整片段边界
- 默认保守（宁可粗一点），编剧有需要再细分

### 3. 状态更新的可靠性

**问题**：依赖 LLM 主动调用 `update_state` 工具，LLM 可能遗漏。

**策略**：
- 编剧在 Prompt 的工具使用指南中明确列出"何时必须更新哪些状态"
- Tool call 有 JSON Schema 校验，格式不会出错
- 引擎检测到一轮结束但未调用 `update_state` 时，在 Debug 面板高亮提示
- 可选 fallback：用一个轻量 LLM 调用从响应中补充提取状态更新

### 4. 可视化编辑器的交互设计

**问题**：如何让非技术用户自然地编辑流程图？

**策略**：
- 参考 Miro/Figma 的节点编辑交互
- 节点点击 → 右侧展开详情面板（Prompt 内容、注入条件等）
- 提供"从文字重新生成"按钮 → 编剧修改源文档后可以重新提取
