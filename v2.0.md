# Interactive Novel Engine v2.0 — Architecture Design

## Design Philosophy

**编剧只写文字。引擎负责理解。**

```
编剧的工作：写 Prompt 文档 + 世界观资料 + 场景设定 — 全部是纯文字
引擎的工作：Agent 从文字中提取结构 → 可视化流程图让编剧确认/调整 → 执行
```

类比编译器：
```
高级语言（编剧的文字）→ 编译（Agent 提取）→ IR（内部数据结构）→ 执行（引擎运行时）
                                ↕
                        可视化 IDE（流程图 + 节点编辑器）
```

编剧不需要学 YAML、模板语法、或任何技术记法。他们的 15,000 字 GM Prompt **已经包含了一切**——阶段地图、NPC 行为规则、节奏预算、收束条件——只是用自然语言写的。引擎通过 Agent 读懂这些文字，自动生成可执行的结构。

## 一、三层架构

```
┌────────────────────────────────────────────────────────────────┐
│  Authoring Layer（编剧接触的层）                                │
│                                                                 │
│  输入：纯文字文档（Prompt、世界观、场景设定）                     │
│  输出：可视化流程图 + 结构确认界面                                │
│  工具：Architect Agent（文字→结构转译）+ Visual Flow Editor      │
├────────────────────────────────────────────────────────────────┤
│  Data Layer — IR（中间表示，对编剧透明）                         │
│                                                                 │
│  Flow Graph（场景流程图数据）                                    │
│  Prompt Segments（拆分后的 Prompt 片段）                         │
│  State Schema（状态变量定义）                                    │
│  Memory Config（记忆策略配置）                                   │
├────────────────────────────────────────────────────────────────┤
│  Runtime Layer（执行引擎）                                       │
│                                                                 │
│  Flow Executor + LLM Client + Memory Manager                    │
│  State Store + Context Assembler + UI Renderer                  │
└────────────────────────────────────────────────────────────────┘
```

## 二、Authoring Layer — 编剧的工作流

### 编剧做什么

编剧的全部工作就是**写文档**，和他们现在做的事情完全一样：

```
module-7/
  docs/
    GM提示词_序章第一章.md        ← 就是现有的 15,000 字文档
    GM提示词_第二章_共鸣池.md
    PC提示词.md
    世界观时间线.md
    共鸣池场景设定.md
    角色设定_女孩.md
```

没有 `flow.yaml`。没有 `state.yaml`。没有模板语法。纯文字。

### Architect Agent 做什么

编剧将文档导入引擎后，Architect Agent（一个 LLM 驱动的分析流程）阅读所有文档，自动提取：

```
输入：编剧的全部文档
  │
  ▼
Architect Agent 分析提取：
  │
  ├─ 1. Flow Graph（流程结构）
  │     "序章：玩家苏醒 → 行走 → 遇到女孩 → ..."
  │     "第二章：进入共鸣池 → 涅瓦 → 卡伦 → ..."
  │     从 GM Prompt 的阶段地图、收束协议中提取
  │
  ├─ 2. State Variables（状态变量）
  │     "girl_language_level: 0-3, GM会更新"
  │     "explored_areas: 列表, 追踪已探索区域"
  │     从 GM Prompt 的 INTERNAL_STATE 定义中提取
  │
  ├─ 3. Prompt Segments（Prompt 拆分）
  │     "这段是通用写作规范 → 可复用"
  │     "这段是第一章阶段地图 → 章节特定"
  │     "这段是共鸣池设定 → 按需注入"
  │     从文档结构和内容语义中拆分
  │
  ├─ 4. Memory Hints（记忆策略建议）
  │     "跨章时需要保留：女孩说过的话、玩家选择、信任等级"
  │     从 GM Prompt 中关于"前章记录继承"的描述提取
  │
  └─ 5. Injection Rules（动态注入规则）
        "当 current_area == 'resonance-pool' 时，注入共鸣池设定"
        "当 girl_language_level >= 2 时，使用不同的行为描述"
        从 GM Prompt 中的条件性描述提取
```

### Visual Flow Editor — 编剧确认/调整的界面

Agent 提取完成后，引擎呈现一个**可视化流程图**：

```
┌─────────────────────────────────────────────────────────────────┐
│  Visual Flow Editor                                              │
│                                                                  │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐                │
│  │ 序章     │────→│ 第一章   │────→│ 记忆压缩 │                │
│  │ 玩家苏醒 │     │ 主循环   │     │ 章节过渡 │                │
│  │ [auto]   │     │ [loop]   │     │          │                │
│  └──────────┘     └──────────┘     └────┬─────┘                │
│                        │                 │                       │
│                   ┌────┴────┐      ┌─────▼─────┐               │
│                   │ 玩家输入│      │ 第二章    │               │
│                   │ [input] │      │ 共鸣池探索│               │
│                   └─────────┘      │ [loop]    │               │
│                                    └───────────┘               │
│                                                                  │
│  ─── 侧边栏 ──────────────────────────────────────             │
│  │ 状态变量:                                                    │
│  │   girl_language_level: 0 (0-3)  ✏️                          │
│  │   explored_areas: []            ✏️                          │
│  │   chapter1_complete: false      ✏️                          │
│  │                                                              │
│  │ Prompt 片段:                                                 │
│  │   📄 GM基础规范 (通用)          ✏️                          │
│  │   📄 第一章阶段地图             ✏️                          │
│  │   📄 共鸣池设定 (按需注入)      ✏️                          │
│  │                                                              │
│  │ 注入规则:                                                    │
│  │   🔗 共鸣池设定 ← 当进入共鸣池时                            │
│  │   🔗 女孩行为v2 ← 当语言等级≥2时                           │
│  ────────────────────────────────────────────                   │
└─────────────────────────────────────────────────────────────────┘
```

编剧可以：
- **确认**：Agent 提取的结构是否正确
- **拖拽**：调整场景顺序、添加/删除节点
- **编辑**：点击节点查看/修改对应的 Prompt 片段
- **连线**：调整分支条件
- **修改状态变量**：重命名、添加、删除
- **调整注入规则**：哪个 Prompt 片段在什么条件下注入

**所有编辑都是可视化操作，不需要写代码或 YAML。**

### 双向工作流

```
编剧修改了文档 → 重新运行 Architect Agent → 流程图更新 → 编剧确认
编剧在流程图上调整 → IR 更新 → 引擎按新结构运行
编剧在流程图上发现问题 → 回到文档修改 → 重新提取 → 循环
```

## 三、Data Layer — IR（中间表示）

IR 是 Agent 提取的结构化数据。编剧不直接编写它，但可以通过可视化编辑器间接修改它。高级用户（程序员）也可以直接编辑 IR。

### Flow Graph

```typescript
interface FlowGraph {
  nodes: FlowNode[];
  edges: FlowEdge[];
}

interface FlowNode {
  id: string;
  type: 'scene' | 'input' | 'compress' | 'state-update' | 'checkpoint';
  label: string;              // 编剧可见的名称，如"序章·苏醒"
  config: NodeConfig;
}

// scene 节点
interface SceneNodeConfig {
  promptSegments: string[];   // 引用的 Prompt 片段 ID 列表
  auto: boolean;              // true = GM 自动生成，不等玩家输入
  model?: string;
  maxTokens?: number;
}

// input 节点
interface InputNodeConfig {
  type: 'freetext' | 'choice';
  choices?: string[] | { fromState: string };  // 静态选项或从状态动态读取
  saveToState?: string;                         // 结果存入哪个状态变量
}

// compress 节点
interface CompressNodeConfig {
  hintPrompt?: string;        // 自定义压缩提示（自然语言）
  pinItems?: string[];         // 压缩时必须保留的信息
}

// state-update 节点
interface StateUpdateConfig {
  updates: Record<string, unknown>;
}

interface FlowEdge {
  from: string;               // 源节点 ID
  to: string;                 // 目标节点 ID
  condition?: string;         // 条件表达式（无条件 = 默认路径）
  label?: string;             // 边上显示的标签
}
```

### 循环的表示

循环在 Flow Graph 中表示为**回边（back edge）**——从循环体末尾指回循环头的边，带有"继续条件"：

```
                    ┌──────────────────────────────┐
                    │                              │
                    ▼                              │
  ┌─────────────────────┐     ┌─────────────┐     │
  │ input: 玩家输入      │────→│ scene: GM回应│─────┘
  └─────────────────────┘     └─────────────┘
                                     │
                          condition: chapter1_complete
                                     │
                                     ▼
                              ┌─────────────┐
                              │ compress     │
                              └─────────────┘
```

回边的 condition 为空 = 无条件循环（受安全上限限制）。
出口边的 condition = 退出循环的条件。

### Prompt Segments

```typescript
interface PromptSegment {
  id: string;
  label: string;                    // "GM基础规范", "第一章阶段地图", "共鸣池设定"
  content: string;                  // 原文内容
  sourceDoc: string;                // 来源文档路径
  sourceRange?: [number, number];   // 来源文档中的行号范围
  role: 'system' | 'context';      // system=始终注入, context=按需注入
  priority: number;                 // 组装时的优先级
  injectionRule?: InjectionRule;    // 何时注入
  tokenCount: number;
}

interface InjectionRule {
  description: string;              // 自然语言描述，编剧可见
  condition: string;                // 编译后的条件表达式
}
```

**Prompt Segment 的来源**：Architect Agent 将编剧的大文档智能拆分为片段。例如：

```
GM提示词_第二章.md (15000字)
  → Segment: "通用写作规范" (2000字, system, always)
  → Segment: "第二章阶段地图" (3000字, system, always when chapter=2)
  → Segment: "涅瓦区域细节" (800字, context, when current_area=neva)
  → Segment: "卡伦区域细节" (600字, context, when current_area=caren)
  → Segment: "自检清单" (500字, system, always)
  → ...
```

编剧在可视化编辑器中可以看到拆分结果，调整边界和注入规则。

### State Schema

```typescript
interface StateSchema {
  variables: StateVariable[];
}

interface StateVariable {
  name: string;
  type: 'number' | 'string' | 'boolean' | 'array' | 'object';
  initial: unknown;
  description: string;            // 自然语言描述
  updatedBy: 'llm' | 'flow' | 'player';  // 谁负责更新
  range?: { min?: number; max?: number };  // 数值范围（如 language_level 0-3）
}
```

### Memory Config

```typescript
interface MemoryConfig {
  contextBudget: number;
  compressionThreshold: number;
  recencyWindow: number;
  compressionHints?: string;       // 自然语言的压缩指导
  crossChapterCarryOver?: string[]; // 跨章需要保留的状态变量名
}
```

## 四、Architect Agent — 从文字到结构

### Agent 的工作流程

```
Step 1: 文档分类
  输入：编剧上传的所有文档
  输出：文档角色标注
    "GM提示词_序章第一章.md" → type: gm_prompt, chapter: 1
    "世界观时间线.md" → type: world_data
    "共鸣池场景设定.md" → type: location_data

Step 2: 状态变量提取
  输入：GM Prompt 中的 INTERNAL_STATE 定义
  输出：StateSchema
  方法：GM Prompt 通常会明确列出需要追踪的变量（MODULE_7 的就列了十几个）

Step 3: 流程结构提取
  输入：GM Prompt 中的阶段地图、收束协议
  输出：FlowGraph
  方法：从"阶段A → 条件 → 阶段B"的描述中提取节点和边

Step 4: Prompt 拆分
  输入：完整的 GM Prompt 文档
  输出：PromptSegment[]
  方法：按语义和功能拆分——通用规范、章节特定、区域特定、条件性内容

Step 5: 注入规则生成
  输入：GM Prompt 中关于"何时使用何种信息"的描述
  输出：InjectionRule[]
  方法：从条件性描述中提取（"当玩家进入共鸣池时，参考以下设定..."）

Step 6: 记忆策略生成
  输入：GM Prompt 中关于记忆/继承的描述
  输出：MemoryConfig
  方法：从跨章继承说明中提取需要保留的信息
```

### Agent Prompt 示例（Step 3: 流程提取）

```
你是一个互动小说引擎的架构分析助手。

请阅读以下 GM 提示词文档，从中提取游戏的流程结构。

输出格式（JSON）：
{
  "nodes": [
    { "id": "...", "type": "scene|input|compress", "label": "...", "config": {...} }
  ],
  "edges": [
    { "from": "...", "to": "...", "condition": "...", "label": "..." }
  ]
}

提取规则：
1. 阶段/章节 → scene 节点
2. "等待玩家输入/回应" → input 节点
3. 阶段之间的转换条件 → edge 的 condition
4. 循环结构（如"重复直到..."）→ 回边
5. 如果文档提到章节过渡或记忆压缩 → compress 节点

文档内容：
[编剧的 GM Prompt 全文]
```

### 多轮精化

Architect Agent 不是一次性提取。它可以多轮交互：

```
Agent: "我从您的第一章 GM Prompt 中提取了 5 个阶段：苏醒、行走、邂逅、陪伴、入睡。
        流程图如下 [显示可视化]。请确认是否正确？"
编剧: "缺少了'坠落'阶段，在邂逅和陪伴之间"
Agent: [更新流程图，添加节点]
编剧: "对了，但坠落阶段有一个分支——玩家可以选择伸手或不伸手"
Agent: [添加 branch 分支]
```

这种对话式的结构精化，比让编剧写 YAML 自然得多。

## 五、Runtime Layer — 执行引擎

### 核心组件

运行时和前面的设计类似，但它消费的是 IR（Agent 生成 + 编剧确认的结构），而不是编剧手写的 YAML：

```
┌─────────────────── UI Layer ───────────────────┐
│  NarrativeView  │  InputPanel  │  DebugPanel   │
│  (流式渲染)      │  (文本/选项) │  (状态/记忆)   │
└───────▲──────────────┬──────────────▲──────────┘
        │        useGameStore         │
        │         (Zustand)           │
┌───────┴──────────────┴──────────────┴──────────┐
│               FlowExecutor                      │
│  读取 FlowGraph → 按节点/边逐步执行            │
│                                                  │
│  scene 节点:                                     │
│    1. ContextAssembler 根据注入规则组装 Prompt   │
│    2. LLM 流式生成 → UI 渲染                    │
│    3. 解析 state block → StateStore 更新        │
│    4. Memory.append → 自动压缩检查              │
│                                                  │
│  input 节点:                                     │
│    1. 通知 UI 等待输入                           │
│    2. 玩家提交 → Memory.append                  │
│    3. 如有 saveToState → StateStore 更新        │
│                                                  │
│  边的遍历:                                       │
│    1. 求值所有出边的 condition                   │
│    2. 走第一个为 true 的边（无条件边 = 默认）    │
│    3. 回边 = 循环                                │
└──────────────────────────────────────────────────┘
│  ContextAssembler  │  LLM Client  │  Memory  │  State  │
└──────────────────────────────────────────────────┘
```

### ContextAssembler — Prompt 组装

ContextAssembler 是运行时中最关键的组件。它根据当前状态，从 Prompt Segments 中选择性组装 LLM 的输入：

```
对每个 scene 节点:

1. 收集该节点关联的 promptSegments
2. 对每个 segment 检查 injectionRule:
   - 无规则 → 始终注入
   - 有规则 → 求值条件，true 才注入
3. 按 priority 排序
4. 加入 state 快照（序列化为 YAML）
5. 加入 memory summaries（压缩摘要）
6. 加入 recent history（近期原文）
7. 按 token 预算裁剪:
   - system segments（不可裁）> memory summary > recent history > context segments
```

### Memory Manager

与前版设计相同：

```
entries: [E0, E1, E2, ..., E_n]     全量记录
summaries: [S0, S1, ...]             分层摘要
watermark: k                          压缩水位线

压缩时：
  保留最近 recencyWindow 条原文
  其余 → LLM 生成摘要（可用编剧提供的 hintPrompt 指导）
  摘要累计过长 → 合并旧摘要
  → O(log N) 增长
```

### State Store

```
KV 存储 + changelog
  get(key) / set(key, value) / update(patch)
  serialize() → YAML（注入 Prompt 用）
  export() / import() — 跨章/存档

状态更新来源：
  1. LLM 输出中的 ```state ... ``` fenced block
  2. FlowGraph 中的 state-update 节点
  3. input 节点的 saveToState
```

## 六、完整工作流示例

### Step 1: 编剧上传文档

编剧将已有的 MODULE_7 文档导入引擎：
- GM提示词_序章第一章_v2_3.md
- GM提示词_第二章_共鸣池_v1_9.md
- PC提示词.md
- 世界观时间线.md
- 共鸣池场景设定.md

### Step 2: Architect Agent 分析

Agent 阅读所有文档，输出：

**提取的状态变量：**
```
✅ current_stage: string      — "从GM Prompt的INTERNAL_STATE中识别"
✅ girl_language_level: 0-3   — "从语言发展阶段描述中识别"
✅ player_behavior_tendency   — "从玩家画像分析描述中识别"
✅ explored_areas: array      — "从共鸣池分区探索追踪中识别"
✅ key_memory_points: array   — "从关键记忆锚点描述中识别"
...
```

**提取的流程图：**
```
[序章·苏醒] --auto--> [行走·荒漠] --auto-->
[邂逅·女孩] --> [玩家输入] --> [第一章循环] ──┐
    ↑                                          │
    └──────── 未完成 ──────────────────────────┘
                   │ 完成
                   ▼
[记忆压缩] --> [第二章·进入共鸣池]
    --> [涅瓦] --> [水阀选择] --> [分支: 卡伦/死海/大泽]
    --> ... --> [马里亚纳·最深处] --> [结局]
```

**提取的 Prompt 片段：**
```
📄 GM通用规范 (所有章节共用, 2100 tokens)
📄 自检清单 (所有章节共用, 800 tokens)
📄 第一章阶段地图 (chapter=1, 3200 tokens)
📄 第一章NPC行为指引 (chapter=1, 1800 tokens)
📄 第二章阶段地图 (chapter=2, 2900 tokens)
📄 共鸣池·涅瓦描述 (current_area=neva, 600 tokens)
📄 共鸣池·卡伦描述 (current_area=caren, 500 tokens)
📄 共鸣池·死海描述 (current_area=dead-sea, 550 tokens)
📄 共鸣池·大泽描述 (current_area=vast-marsh, 520 tokens)
📄 共鸣池·马里亚纳描述 (current_area=mariana, 680 tokens)
📄 世界观时间线 (按需, 1200 tokens)
```

### Step 3: 编剧在可视化编辑器中确认

编剧看到流程图和提取结果，进行调整：
- "坠落阶段缺少了" → 在图上添加节点
- "共鸣池的探索顺序不应该是固定的" → 将线性连接改为分支循环
- "涅瓦的描述应该拆成两段，一段是物理空间，一段是女孩的行为" → 点击片段，调整拆分

### Step 4: 确认后运行

编剧点击"开始"，引擎按确认后的 FlowGraph 执行。

## 七、与 v1 的对比

| 维度 | v1 (当前仓库) | v2 |
|------|-------------|-----|
| **编剧写什么** | JSON 表单 | 纯文字文档（他们已经在写的） |
| **结构化工作** | 编剧手动完成 | Agent 自动提取 + 编剧可视化确认 |
| **游戏循环** | 引擎硬编码 9 步 | FlowGraph（可视化编辑） |
| **Prompt 管理** | 固定 JSON 字段 | 智能拆分为片段 + 按需注入 |
| **记忆** | 单层关键词匹配 | 分层压缩 + O(log N) |
| **Token 管理** | 无 | 自动预算分配 |
| **编剧门槛** | 需要理解 JSON/GOAP | 只需要会写文字 + 看懂流程图 |
| **灵活性** | 只能做 GOAP 模拟 | Agent 能从任意文字提取任意结构 |

### 本质差异

```
v1:  编剧 → [手动转译为JSON] → 引擎执行（编剧做翻译工作）
v2:  编剧 → [Agent 自动转译] → [编剧可视化确认] → 引擎执行（Agent做翻译工作）
```

**编剧的认知负担从"学习引擎的数据格式"变为"确认 Agent 的理解是否正确"。** 后者是编剧擅长的事。

## 八、关键技术挑战

### 1. Architect Agent 的提取质量

**问题**：Agent 能否准确从 15,000 字的自然语言中提取出正确的流程结构？

**策略**：
- 分步提取（先提变量、再提流程、再拆 Prompt），每步用专门的 prompt
- 多轮确认——Agent 提取后，编剧在可视化界面上校正，Agent 学习修正
- 编剧的文档写得越结构化（如 MODULE_7 的 GM Prompt 已经有清晰的阶段地图），提取越准确
- 对于模糊的部分，Agent 标记"不确定"让编剧手动确认

### 2. Prompt 拆分的粒度

**问题**：把一个 15,000 字文档拆成多少个片段？太粗 = 无法按需注入；太细 = 上下文断裂。

**策略**：
- 以文档自身的 Markdown 标题结构为初始拆分点
- Agent 根据语义相关性建议合并或再拆分
- 编剧在可视化界面中可以调整片段边界（拖拽分割线）
- 默认保守（宁可粗一点），编剧有需要再细分

### 3. 状态更新的可靠性

**问题**：依赖 LLM 在输出末尾附加 state block，LLM 可能遗漏。

**策略**：
- 在 Prompt 中用醒目格式强调（MODULE_7 的 GM Prompt 已经这么做了）
- 引擎检测到缺失时，用一个轻量 LLM 调用从响应中提取状态更新（fallback）
- 在 Debug 面板中高亮显示状态未更新的轮次，方便发现问题

### 4. 可视化编辑器的交互设计

**问题**：如何让非技术用户自然地编辑流程图？

**策略**：
- 参考 Miro/Figma 的节点编辑交互，编剧群体对这类工具有认知基础
- 节点点击 → 右侧展开详情面板（Prompt 内容、注入条件等）
- 连线拖拽 → 自动弹出条件编辑器（下拉选状态变量 + 比较 + 值）
- 提供"从文字重新生成"按钮 → 编剧修改源文档后可以重新提取
