## 三个严重程度不同的缺陷

### 🔴 缺陷 1：pin_memory 核心作用缺失（章节内）

- **现象**：编剧在 prompt 里教 LLM 调 `pin_memory`，LLM 调了也存进去了，但**下一轮 LLM 在 MEMORY section 里看不到这些 pinned 内容**
- **根源**：`context-assembler.ts:170-184` 只调 `memory.getInheritedSummary() + getSummaries()`，漏掉 `memory.getPinnedEntries()`
- **当前为什么凑合能跑**：`recencyWindow: 100` + MODULE7 约 60 轮 = 所有 entries 都在 recent messages 里，"pin" 和"不 pin"对 LLM 视野没差别
- **破绽场景**：
  - 一旦 playthrough 超过 100 轮，老的 pinned 内容从 recent 窗口滑出就永远消失
  - 一旦 recencyWindow 调小（比如为了省 token 设 20-30），pin 的内容立刻看不见
- **修复成本**：10 行代码，在 MEMORY section 组装时加 `memory.getPinnedEntries()` 的拼接

### 🟡 缺陷 2：章节继承定义了但从不触发

- **现象**：`chapter-transition.ts` 有完整的 `executeChapterTransition` 实现，会把上一章的 summaries + pinned 打包成 inherited summary 传给新章节
- **根源**：**没有任何调用方**。MODULE7 第一章→第二章切换时，memory 继续线性累积，inherited summary 永远为空
- **当前 MODULE7 能跑的原因**：第二章 prompt 自己硬编码了"玩家已接受女孩成长"等前提，不依赖引擎继承
- **破绽场景**：
  - token 消耗偏高——第二章还背着第一章全部 history
  - 如果 playthrough 长度超出 recencyWindow，压缩出来的 summary 混着两章内容不分
  - 未来多章节剧本会恶化
- **修复成本**：中等。需要在 game-session 里监听 `update_state({chapter: N+1})` → 触发 `executeChapterTransition`
- **建议时机**：试玩启动后第一个迭代，不拦上线

### 🟢 缺陷 3：compressFn 是拼接不是压缩

- **现象**：`game-session.ts:349-354` 的默认 compressFn 只是把 entries 每条截到 200 字再 join，不是 LLM 真压缩
- **后果**：所谓 "summary" 实际是"短版原文"，压缩比 1-3× 不是应有的 10-20×
- **当前为什么不成问题**：recencyWindow=100 + MODULE7 约 60 轮 = **永远不会触发 needsCompression**
- **破绽场景**：支持 200+ 轮长剧本、或把 recencyWindow 降到 20-30 时会看到 summary 把 token 吃爆
- **修复成本**：中等。需要在 compressFn 里调一次 LLM
- **建议时机**：等真的需要跑长剧本时再做，**试玩期不用动**

## 与 R4（player_attitude_signals）的交互

Round 1 · R4 采纳的方案：在 state 里加 `player_attitude_signals` 数组，LLM 每轮判断玩家态度记一条。

**这条 state 的可靠性依赖于 recent history 注入**（现在 ✅ 完整工作）+ state section 组装（现在 ✅ 完整）。所以 R4 不受审计缺陷影响。

**但如果缺陷 1 不修**：未来如果编剧把 player_attitude_signals 之类的"重要历史"用 pin_memory 保留，会失效。这是另一个需要记住的事。
